## Security
<!-- Change exposes a new user-controlled path to subprocess execution and new captured-content handling -->
- [ ] CHK-059 Session input validation before subprocess: `CapturePaneByWindow` and `handleWindowCapture` both call `validate.ValidateName` on `session` before any tmux subprocess is spawned; malicious payloads like `s1; rm -rf /` or `s$(rm -rf /)` return 400/validation error with no subprocess started (test in `windows_test.go`).
- [ ] CHK-060 Window index validation: `parseWindowIndex` is used on the path segment before constructing the `{session}:{index}` target, rejecting non-integer and negative inputs.
- [ ] CHK-061 `lines` parameter bounds: clamped to `[1, 100]` in the handler, rejected outside `[1, 100]` in `CapturePaneByWindow`, guarding against DoS via huge capture requests.
- [ ] CHK-062 Pane context timeouts enforced: every `CapturePaneByWindow` call sites pass a bounded `context.WithTimeout` (3s in both SSE enrichment and the HTTP handler); no unbounded ctx usage.
- [ ] CHK-063 No shell string construction for tmux: tmux argv is a `[]string` slice passed to `exec.CommandContext`; no `sh -c`, no format-string shell, no `exec.Command` without a context.
- [ ] CHK-064 ANSI stripping does not execute escape sequences: `StripANSI` operates as a pure regex replace on the string; output is plain text only — no rendering, no `dangerouslySetInnerHTML`, no unescaped HTML injection in the sidebar peek block (peek lines rendered as React text children).
- [ ] CHK-065 No zombie subprocesses on timeout: `exec.CommandContext` cancellation terminates the tmux child process on deadline exceeded (verified by the tmux_test.go timeout scenario).
- [ ] CHK-066 Captured content size bound: enrichment fixes lines=3; handler caps at 100; combined with per-capture timeout this bounds both memory and wall-clock cost per request and per tick.
