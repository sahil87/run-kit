RUN-KIT · DESIGN STUDY · LABEL PICKER

Markers, a shade axis, and more easter eggs

Three studies on the combined Label picker. Every swatch and row tint below is computed live with the real pipeline from themes.ts — OKLCH hue families adapted to default-dark's mean L/C, saturated ×1.5, blended at 14/22/40%.

[ STUDY 1 · MARKERS — DASHED + THICK ]

The stripe vocabulary grows from 3 to 5 states, ordered thin→heavy. Both new rhythms keep the seam-free rule: pattern periods must divide gcd(24, 36) = 12px so stacked rows merge continuously (the lesson that made dotted a gradient, not a border).

[ STUDY 2 · COLOR — NORMAL / DARK SHADE AXIS ]

Each family renders at the theme's mean lightness L. The shade axis keeps today's color as-is — normal = mean L, dark = L−0.14, chroma-reduced into gamut — so hue identity stays stable and every existing colored row maps onto the normal shade untouched. 10 families become 20 values. Three ways to fit them:

VARIANT A · SHADE ROWS (10×2)
Family = column, shade = row. The axis reads instantly, but the popover doubles in width and the marker column can't row-pair with 2 color rows.
VARIANT B · SPLIT SWATCH (10, HALVES)
Same footprint; top half = normal, bottom = dark. But each click target is 18×11px — fails the coarse-pointer budget, and halves read as the existing base/selected preview split. Weakest option.
VARIANT C · PAIRED GRID — CHOSEN ← click swatches & markers
20 swatches 4-wide = 5 color rows (+ Clear = 6), pairing 1:1 with the 6 marker cells. And the marker cells are now live row previews: pick any color and they repaint with its row tint, guarded stripe color, and paired texture — flowing dashes, hazard weave, scanlines.
DOES THE AXIS SURVIVE THE TINT BLEND? · SELECTED ROWS, NORMAL VS DARK
At 14–40% blend into a dark background the shade gap compresses — the axis mostly survives in the selected tint and the gutter stripe, less in the resting tint. If shades read too close, boost the dark variant's blend ratio a few points or let shade ride the stripe alone.
  1. Storage forces the vocabulary migration. The legacy numeric descriptors ("4", "1+3") have no shade slot. Either extend them ("4:d") or finally store family names ("blue-dark") — the backend ValidateColorValue changes either way, ending the zero-migration run.
  2. Slate is a degenerate case. Near-neutral chroma means its light/dark variants are just two grays — arguably fine (it doubles as an archive shade ramp), but worth deciding deliberately.
  3. Selection highlight collision. Adjacent light/dark cells of one family look similar at 18px; the selected ring must be unambiguous. Variant C keeps pairs adjacent so the ring position disambiguates.
[ STUDY 3 · ROW TREATMENTS — THE PAIRING TABLE ]

The working pairing. Semantics: dashed = working (agent active), thick = completed, double = different (what "different" means is the user's call); none / dotted / solid are entirely free per-user conventions. All markers and row colors are user preferences — what the user thinks about a task — deliberately distinct from the DERIVED status-dot states; the two axes never mix.

MARKER dotteddashed working soliddouble different thick completed
BACKGROUND —data rain always-on —scanlines hazard wedge static

Discipline unchanged — background-position/opacity only (never transform, the drag-ghost rule), static under prefers-reduced-motion, colored through --rk-marker-color, all rhythms on a 12px module so 24/36px rows weld. One treatment moves at rest: dashed's DATA RAIN, always-on — the stripe stays static while two sparse dash tracks stream left→right ("working" is inherently live; the thinned rain proved quiet enough after watching both gatings). Double's scanline crawl + band stays selection-gated. The hazard wedge never moves in any state — completed is the most resting state, so it gets the quietest treatment: the weave fades out by a third of the row, a taped-off corner rather than wallpaper. Picker preview cells never animate.

One design flag: if dashed carries machine state (busy), it stops being a user-owned label — the busy fact already flows through @rk_agent_state, so the always-on stream could be derived from agent state instead of set by hand, keeping the marker axis purely yours (Constitution X: derivation wins). Worth deciding before wiring.
