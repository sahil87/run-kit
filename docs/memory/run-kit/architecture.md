# run-kit Architecture

## System Overview

run-kit is a web-based agent orchestration dashboard. In production, a single Go binary runs as a daemon in a dedicated tmux session:

1. **Go backend** (`app/backend/`, default port 3000) — single binary serving REST API, SSE, WebSocket terminal relay, and SPA static files on one port. Cobra CLI with subcommands: `serve` (default, with `-d`/`--restart`/`--stop` daemon flags), `update` (alias: `upgrade`), `doctor`, `status`, `context`, `init-conf`, `riff`. Version info via `--version`/`-v` global flag (Cobra built-in)

In development, `just dev` runs two concurrent processes:
- Vite dev server (`:RK_PORT`, default 3000) — HMR, proxies `/api/*`, `/relay/*`, and `/proxy/*` to Go backend
- Go backend (`:RK_PORT+1`, default 3001) — API, WebSocket relay, SPA static serving

Configuration via env vars: `.env` (committed) defines `RK_PORT` and `RK_HOST`, `.env.local` (gitignored) for overrides. Go backend and Vite both read `RK_PORT`/`RK_HOST` directly — no intermediate `BACKEND_*` vars. In dev mode, `dev.sh` is the translation layer: it passes `RK_PORT+1` to the Go backend and `RK_PORT` to Vite. `dev.sh` accepts `--port` for ad-hoc overrides. Tmux config defaults to `~/.rk/tmux.conf` (scaffolded by `rk init-conf`), overridable via `RK_TMUX_CONF` env var. The config is embedded in the binary via `go:embed` and written to disk by `init-conf`. All consumers use `tmux.ConfigPath()` getter — no direct env var reads elsewhere. User extensions go in `~/.rk/tmux.d/*.conf` — a drop-in directory sourced via `source-file -q` at the end of the default config. Files load in lexicographic order (numeric prefixes control precedence). The `-q` flag silences errors when the directory is empty or missing.

run-kit connects to **one tmux server at a time**, determined by the URL path segment. The URL hierarchy is `/$server/$window` (since `260529-jad6-window-api-stability`) — the server name is always in the URL, making URLs fully shareable and self-contained. The window segment is the **bare numeric part** of the stable tmux **window ID** (`@N` sans `@`, e.g. `/$server/0`, since `260703-8mpy-numeric-window-url`) — still NOT the mutable window index (the distinction from `260529-chgz-window-id-routing` survives: the number in the URL is the ID's digits, restored to `@N` by the route's parse, never the positional index). The API/relay layers still address a window by the `@N` form (e.g. `/api/windows/%40N`, `/relay/%40N` — unchanged); only the page URL segment drops the `@`. `$session` was **dropped from the route shape** by `260529-jad6` — `@N` is server-global so the session is redundant in the URL; it is derived server-side / from the active window's SSE snapshot wherever a session name is needed for display. Old 3-segment `/$server/$session/$window` URLs are a hard break (no redirect shim). The active server is sent as a `?server=` query parameter on every API request (SSE, REST, WebSocket) by the frontend's `withServer()` helper. The backend is stateless — it defaults to the `default` tmux server when the parameter is absent. Server discovery scans `/tmp/tmux-{uid}/` for socket files. The frontend persists the last-used server in localStorage (`runkit-server`) as a convenience for the server list page, but the URL is always the source of truth. Server lifecycle: create (via server list page or command palette), kill (`tmux kill-server`), switch (navigate to `/$newserver`).

The pane-boards view (`/board/$name`) is a deliberate exception: it is a root-level peer to `/$server` (NOT under `/$server/...`) because boards aggregate windows across multiple tmux servers. It does not render the server-scoped AppShell — see `## Boards Feature` below.

## Data Model

**No database.** State derived at request time from:
- **tmux server** — `tmux list-sessions`, `tmux list-windows` via `internal/tmux/tmux.go`. Project roots derived from window 0's `pane_current_path`. Read-path functions (`ListSessions`, `ListWindows`, `ListServers`) accept `context.Context` from HTTP handlers for request cancellation propagation. Mutation functions (Create/Kill/Rename) use `context.Background()` — user-initiated actions complete regardless of client disconnect
- **`run-kit.yaml`** — per-project config file at the git root of a session's working directory. `config.FindGitRoot()` (walks up from `pane_current_path` looking for `.git`) has live callers (`internal/sessions`, `ProjectRoot`). **The `session_color` path is DEAD** — `config.ReadSessionColor()` / `WriteSessionColor()` / `parseSessionColor` (still typed `*int`) have ZERO non-test call sites: session color migrated to the tmux `@session_color` option (below) before this change, and `260615-6rnr-expand-swatch-palette-blends` did not revive them (the int→string color migration made their staleness conspicuous — flagged as deletion candidates, left for a human). `run-kit.yaml` no longer stores any color
- **tmux `@color` / `@session_color` user options (color storage, since `260615-6rnr`)** — per-window color in the `@color` user option (`#{@color}` is field 8 of the `ListWindows()` format string); per-session color in the `@session_color` option (field 5 of `ListSessions()`'s format string — a distinct name from window `@color` to avoid tmux option inheritance). Both are ephemeral (survive session lifetime, not tmux-server restarts). The stored value is a **string color-value descriptor** — `"4"` for a single ANSI index or `"1+3"` for a two-hue blend (NOT a bare integer, though a legacy bare int is still accepted on read). Window color set via `SetWindowColor`/`UnsetWindowColor` (`set-option -w -t {windowID} @color {value}` / `-wu`); session color set via `SetSessionColor(session, value, server)` / `UnsetSessionColor` (`set-option -t {session} @session_color {value}` / `-u`). Both parse the raw option through `validate.NormalizeColorValue` (accepts legacy int OR descriptor, normalizes to the canonical string, drops on malformed) into `WindowInfo.Color *string` / `SessionInfo.Color *string` (nil when unset/malformed); `SessionInfo.Color` surfaces to the SSE snapshot as `ProjectSession.SessionColor` in `internal/sessions`. Both `-t` targets are self-contained (window ID; session name) — no `session:index` string
- **tmux `@rk_session_order` server-scoped user option** — sidebar session render order stored as a JSON-encoded array of session names on the tmux *server* (not per-window). Survives the same lifetime as the tmux server (lost on tmux server death, NOT on rk-go restart per Constitution VI). Read by `tmux.GetSessionOrder(ctx, server)` — treats unset / no-server / `failed to connect` as empty slice; surfaces other errors (including JSON decode) as wrapped Go errors. Written by `tmux.SetSessionOrder(ctx, server, order)` — `nil` slice normalized to `[]` so round-trips are lossless. Both functions go through `tmuxExecRawServer` with `context.WithTimeout(ctx, TmuxTimeout)` (10s). The constant `tmux.SessionOrderOption = "@rk_session_order"` is the canonical option name
- **fab pane map** — `internal/sessions` invokes the `fab` router on PATH via `exec.CommandContext(ctx, "fab", "pane", "map", "--json", "--all-sessions")` with `cmd.Dir = repoRoot` so the router can resolve the project's `fab_version` from `fab/project/config.yaml`. Returns the **fab tier proper** per window — change name, pipeline stage, display-state. Since `260705-dmex-generic-agent-state-tier` the join **no longer consumes** `agent_state`/`agent_idle_duration`/`pr_url`/`pr_number` from the map: agent lifecycle state moved to the `@rk_agent_state` pane option (Tier 2, read natively by `internal/tmux` — see [agent-state](agent-state.md)) and PR links are derived server-side from the pane's branch (see § Branch→PR Derivation), so any such keys still emitted by fab are simply ignored. `fab pane map` emits a stable tmux **pane ID** (`"%N"`) per entry, so `keyPaneEntries` builds a fetch-time lookup keyed by that pane ID (one entry per pane, **no** window-level dedup at fetch time — window membership is only knowable against a fresh snapshot). `FetchSessions` then joins by fresh **pane membership** via the pure `joinPaneMapByWindow`: for each window of the live `WindowInfo` snapshot it walks the window's panes and looks each `PaneID` up in the map, selecting **Change > first-seen** among a window's candidate panes (a change-bound entry wins, else the first-seen candidate in pane order — the prior fetch-time dedup semantics, moved to join time; the former AgentState tiebreak arm is gone because agent state no longer rides the map), and lands the winner in `enrichByWindowID[window.WindowID]`. Because the pane ID travels with its window across `swap-window`/cross-session move exactly like the window ID, a **stale** cached map (≤5s TTL) can never misattribute one window's fab state to another across a reorder/move — an entry can only attach to the window that actually contains its pane; at worst a pane absent from the fresh snapshot contributes nothing (since `260713-d07t-pane-map-join-by-pane-id`, which replaced the former `dedupEntries` `session:index` map + `index→windowID` re-key join that glued fab state to whichever window sat at an old index for the stale-cache window). Entries with an empty `Pane` field (hypothetical older fab JSON) fall back to the legacy `session:windowIndex` key (`paneMapKey`), joined once per window when no pane-ID candidate matched (shapes cannot collide — a real pane ID begins with `%`). Results cached at package level with 5s TTL (`fetchPaneMapCached`) — double-check pattern prevents thundering herd, stale cache preserved on fetch error for graceful degradation

### Performance Caching

Two in-memory caches reduce subprocess spawning on the SSE hot path (justified per code-quality.md performance carve-out — not general-purpose caching):

| Cache | Location | TTL | Key | Purpose |
|-------|----------|-----|-----|---------|
| SSE session cache | `sseHub.cache` (`api/sse.go`) | 500ms | server name | Skip `FetchSessions()` subprocess calls when data is fresh |
| Pane-map cache | package-level (`internal/sessions`) | 5s | tmux server label | Skip `fab pane map` subprocess calls between SSE ticks |

Both caches use `sync.RWMutex` for thread safety. The SSE hub's existing JSON diff-check (`previousJSON`) remains as a separate concern — the session cache prevents redundant subprocess spawning, while the diff prevents redundant SSE broadcasts

## API Layer

All endpoints served by the single Go binary on one port. POST-only mutations with path-based intent (no multiplexed action field). **Uniform HTTP verb (constitution §IX, enforced since `260529-jad6`)**: every mutating endpoint uses `POST`; there are zero `PUT`/`PATCH`/`DELETE` route registrations and the CORS `AllowedMethods` allowlist is exactly `[GET, POST, OPTIONS]`. `260529-jad6` migrated the five remaining `.Put(` routes to `POST`: `sessions/order`, `settings/theme`, `settings/server-color` (verb-only), plus `windows/{id}/url` and `windows/{id}/type` (folded into `POST /options`, removed rather than migrated). Endpoint semantics that would conventionally map to other verbs (e.g. partial update on `/options`) are expressed via the path and a documented body contract.

**Window addressing (since `260529-chgz-window-id-routing`)**: routes that act on a *specific existing window* are keyed by the stable tmux **window ID** under `/api/windows/{windowId}/...` (kill, move, move-to-session, rename, options, keys, select, split, close-pane). The `{windowId}` path param is validated via `validate.ValidateWindowID` (`^@[0-9]+$`) by the `parseWindowID(r) (string, bool)` helper in `api/windows.go`, which delegates to the shared `decodeWindowID(r) (string, bool)` helper (since `260529-jad6` — `url.PathUnescape` + `ValidateWindowID` in exactly one place; `handleRelay` calls the same helper, eliminating the copy-paste drift that caused bug #205). `400` on malformed ID before any tmux call. A window ID is a server-global, self-contained tmux target, so the session name drops out of these paths. Routes that *create* a window or enumerate a session's windows stay session-scoped (`POST /api/sessions/{session}/windows`, session enumeration) — there is no window ID at create time.

**`broadcastServerOrder` — server-global fan-out (vs. per-server `broadcastSessionOrder`)** (since `260705-bpnr-server-tiles-drag-reorder`): the SSE hub gains `broadcastServerOrder(order []string)` (`api/sse.go`) alongside the existing `broadcastSessionOrder(server, order)`. The distinction is load-bearing and deliberate: **session** order is a per-server concern (which server's sidebar reorders), so `broadcastSessionOrder` fans only to that server's clients and caches per-server in `previousOrderJSON[server]`. **Server** rank order is a HOST-global concern — a client viewing one server, or NONE (the bare `/` Cockpit on the `?metrics=1` sentinel stream), still needs to re-sort its whole server list — so `broadcastServerOrder` fans to EVERY client across every `h.clients` key (exactly like the `metrics`/`services` broadcasts) and caches in a SINGLE `cachedServerOrderJSON` slot replayed to every new client in `addClient` (NOT gated on `c.server`). It normalizes a `nil` order to `[]` so the cached JSON is always `"[]"`, never `"null"`. This is why the design chose a fresh server-global event rather than reusing the per-server `session-order` shape — the latter would neither reach `?metrics=1` clients nor fit a host-global cache.

**`broadcastBoardOrder` — server-global fan-out (vs. per-server `broadcastBoardChanged`)** (since `260708-a2qd-board-list-reorder`): the SSE hub gains `broadcastBoardOrder(order []string)` (`api/sse.go`), a verbatim structural mirror of `broadcastServerOrder` — a board is an emergent cross-server aggregate of `_rk-pin-*` sessions with no owning tmux server, so board **display order** is a HOST-global concern (exactly like server rank order, and unlike the per-server `board-changed` pin/unpin/reorder event which fans only to one server's clients). It fans `event: board-order` to EVERY client across every `h.clients` key (incl. the `?metrics=1` sentinel) and caches in a SINGLE `cachedBoardOrderJSON` slot replayed to every new client in `addClient` (NOT gated on `c.server`), normalizing a `nil` order to `[]` so the cached JSON is always `"[]"`, never `"null"`. Payload `{"order":[...]}`. Fired synchronously by `handleBoardOrderPost` after the `settings.SetBoardOrder` write; it deliberately does NOT reuse the per-server `board-changed` event (which is scoped to a tmux server and uncached), since an order write is host-global.

**`event: version` (set-once cached slot) + `event: update-available` (broadcast + cached replay)** (since `260713-4zap-update-notify-one-click-upgrade`): two more server-global cached slots on `sseHub`, both delivered to EVERY client across every `h.clients` key including the `?metrics=1` sentinel (fan-out identical to `server-order`/`board-order`), replayed on connect from `addClient` (after the server-order/board-order sends, both gated on a non-empty cached string; NOT gated on `c.server`). They differ in whether they broadcast:

- **`event: version`** — the RUNNING daemon version (ldflags `main.version`, sentinel `"dev"` for local builds), payload `{"version":"0.5.3"}`, cached in `cachedVersionJSON`. Seeded ONCE via `sseHub.setVersion(version)` (called from `Server.SetVersion` after `NewRouterAndServer` — see § API Server version/checker wiring); the running version cannot change for the process lifetime, so there is **NO broadcast path** — it is a **replay-on-connect-only** slot (an empty version is ignored, leaving the slot empty → no event). This is the sole version-exposure mechanism — deliberately no `GET /api/version` (minimal surface §IV); the SSE connect IS the moment the client needs it, and the reconnect after a daemon restart is exactly the signal the frontend reload guard keys on.
- **`event: update-available`** — a qualifying pending update, payload `{"current":"0.5.3","latest":"0.6.0"}`, cached in `cachedUpdateAvailableJSON`. `broadcastUpdateAvailable(current, latest)` (pattern: `broadcastBoardOrder`) marshals the payload, sets the cache, and fans the `event: update-available` frame to every client under `h.mu.Lock()` (drop-with-`slog.Warn` on a full channel). Empty until a check qualifies. Invoked via the `Checker.OnQualify` callback wired in `serve.go` (see below), so a later NEWER release re-broadcasts (the checker's re-fire gate) and a late-connecting client gets the cached slot on connect.

### Riff Engine Injection + Repo-Root Derivation (`260713-sbk1-web-spawn-agent`)

The `POST /api/riff` + `GET /api/riff/presets` handlers surface the extracted `internal/riff` spawn engine (full engine in [rk-riff](rk-riff.md)). Two seams are load-bearing:

**`RiffEngine` interface — a DEDICATED dependency, NOT folded into `TmuxOps`** (`api/router.go`): `type RiffEngine interface { Spawn(ctx, riff.Options) (riff.Result, error) }`, injected onto the `Server` struct as a nil-safe `riff RiffEngine` field mirroring the `metrics`/`services`/`prStatus` collector-injection pattern. Production wiring is `prodRiffEngine{}` (delegates to `riff.Spawn`); tests inject a focused mock via the new `NewTestRouterWithRiff(logger, sf, ops, engine, hostname)` constructor. **Why a separate field rather than a `TmuxOps` method**: `TmuxOps`/`mockTmuxOps` are shared by every handler test, so a spawn method there would force a mock method into dozens of unrelated tests; the dedicated field keeps the blast radius to the riff handler + its own mock. `NewTestRouter` leaves `riff` nil (riff-unrelated tests never reach it).

**Repo-root derivation from the target session's active pane** (`deriveRepoRoot` in `api/riff.go`, shared by both handlers): `ListWindows(ctx, session, server)` → the active window (`IsActiveWindow`, else the first) → that window's active pane (`PaneInfo.IsActive`, else the first pane, else the window's `WorktreePath`) → `Cwd` → `config.FindGitRoot`. It returns `(root, cwd, err)` so a non-repo caller can name the offending cwd in its `400`. The three no-root outcomes are distinguished: a derived-but-non-repo cwd, a session with no window/pane to derive a cwd from (the practical nonexistent-session path — the real `ListWindows` swallows its tmux error into an empty result), and a tmux-read failure (surfaced as a `400` "does it exist?" message, since a named-session read failure is most likely a gone session, client-correctable). This uses the existing `ListWindows` surface — no new tmux primitive — and is session-scoped (not window-scoped) because the endpoint's spawn target is a session, not a window.

The `SessionProvider` sets the getter on mount. No multiplexed `action` field — each mutation is a separate function with its own URL path.

**Request deduplication**: A module-level `deduplicatedFetch` wrapper maintains a `Map<string, Promise<Response>>` of in-flight GET requests. When a GET to a URL already has an in-flight promise, the existing promise is returned (with `Response.clone()` so each caller can independently consume the body). POST/PUT requests always make fresh `fetch()` calls. Promises are cleaned up via `.finally()` on resolve or reject. All GET functions (`getHealth`, `getSessions`, `getDirectories`, `listServers`, `getKeybindings`, `getThemePreference`) use `deduplicatedFetch`; all mutation functions use plain `fetch`. Deduplication key is the full URL string (after `withServer()` appends `?server=`), so requests scoped to different tmux servers are not incorrectly merged.

## Terminal Relay

WebSocket endpoint at `/relay/{windowId}?server=runkit|default` on the same port as the API — no separate relay port. Uses `gorilla/websocket` for WebSocket handling and `creack/pty` for PTY allocation. Implementation in `app/backend/api/relay.go`. The `server` query param determines which tmux server to attach to (defaults to `runkit`). The `{windowId}` is validated via `validate.ValidateWindowID` before the WS upgrade — a malformed ID responds `400` (no upgrade, no tmux call).

Per connection (since `260602-qn62-move-based-board-pin-sessions`, the relay attaches **directly** — no ephemeral):
1. **Validates the window ID** via the shared `decodeWindowID(r)` helper (`url.PathUnescape` + `validate.ValidateWindowID`) — a malformed ID is a `400` before the WS upgrade or any tmux call (the same helper the REST path uses, so they cannot drift — bug #205)
2. **Resolves the owning session from the window ID** via `s.tmux.ResolveWindowSession(ctx, server, windowID)` (5s timeout) — the result is the ONE session that owns the window: a normal home session OR a board pin-session (`_rk-pin-*`). Since the move-based model removes window sharing, an O(1) targeted `display-message -t <windowID> -p '#{session_name}'` resolves that single owning session unambiguously (restored from the prior O(n) `list-windows -a` scan by `260609-enic-restore-display-message-resolve-window` — a relay-connect latency regression). If resolution fails or returns empty (unknown window ID), the relay closes the WebSocket with code `4004` ("Window not found")
3. **Selects the window** on its real session via a direct bare `s.tmux.SelectWindow(windowID, server)` (`select-window -t @N`) so the attach renders the right window. For a pin-session this is a no-op (sole window always active); for a home session it mutates the real session's single active-window pointer — the accepted multi-client collision tradeoff
4. Waits up to 5s for the first `{"type":"resize"}` message to start the PTY at the correct dimensions (falls back to 80×24)
5. Best-effort `tmux.ReloadConfig(server)` so true-color terminal-overrides are active even on a foreign server, then spawns `tmux [-L server] attach-session -t <session>` via `creack/pty` (`forceTERM` sets `TERM=xterm-256color`). Attaches **directly to the resolved owning session** — no ephemeral, no defer-kill; the session is durable and owned by tmux
6. Relays I/O between WebSocket and pty (goroutine for pty→WS, main loop for WS→pty); handles resize via `pty.Setsize`
7. On disconnect: `sync.Once` cleanup cancels context, closes PTY, kills the attach process. PTY reader goroutine calls `cleanup()` (not `conn.Close()`) on read failure — eliminates concurrent WebSocket close race. WebSocket closed only by `defer conn.Close()` on the main goroutine

**Why no ephemeral** (the `260602-qn62` simplification): the old per-WebSocket grouped ephemeral (`rk-relay-*`) existed only to give each connection its own active-window pointer (a tmux session has exactly ONE, shared across attachments) so board panes pointing at different windows of one session wouldn't collide. The move-based board model removes window *sharing* — each board pane's window now lives in its own single-window pin-session whose sole window is permanently active — so the isolation layer has no purpose and was deleted: `newEphemeralRelayName`, `tmux.NewGroupedSession`, the `@rk_owner_pid` stamp (`SetSessionOwnerPID`/`GetSessionOwnerPID`/`OwnerPIDOption`), the ephemeral `defer KillSessionCtx`, the scoped `SelectWindowInSession`-on-ephemeral, `RelaySessionPrefix`, and `ListRawSessionNames` are all gone. **Accepted tradeoffs**: multi-client navigation now mutates a shared home session's active window (two tabs / a navigating viewer yank each other); a no-op for pin-sessions. See `tmux-sessions.md` § Terminal Relay and § Pin Sessions.

**No startup sweep** (since `260602-qn62`): `cmd/rk/serve_sweep.go` (`sweepOrphanedRelaySessions`/`pidAlive`/`relayOwnerIsDead`) is **deleted** and unwired from `serveCmd.RunE`. Relay ephemerals no longer exist, and board pin-sessions (`_rk-pin-*`) are PERSISTENT across rk restarts (a valid state, not an orphan — Constitution VI), so there is no in-server session class to reap. The `_rk-ctl` anchor + `exit-empty off` backstop are untouched.

Client-side WebSocket reconnection: exponential backoff (1s, 2s, 4s, 8s, 16s, max 30s) on unexpected close. Shows `[reconnecting...]` in terminal. Re-sends resize on successful reconnect. Skips reconnect on component unmount. On close code `4004` (session/window not found): shows `[session not found]` and navigates to `/` instead of reconnecting. Terminal page connects via `ws://${location.host}/relay/${encodeURIComponent(windowId)}?server={runkit|default}` — same host, window ID (`@N`) in the path, server param from session metadata. The session name no longer appears in the relay URL.

## Daemon Lifecycle

The Go binary manages its own daemon lifecycle via the top-level `rk daemon` subcommand tree (sibling of `rk serve`, not nested under it). The daemon runs in a dedicated tmux server (socket `rk-daemon`, session `rk-daemon`, window `serve`) — separate from the `runkit` server used for agent sessions. The session name matches the socket name so the combined exact-match target (`=rk-daemon:=serve`) cannot collide with user-facing sessions via tmux prefix matching. `LegacySessionName = "rk"` is consulted only by `Stop`/`Restart` (via `runningSessionCtx`, which checks `SessionName` first then falls back to the legacy name) so a daemon started by a pre-rename binary can still be stopped cleanly during the first upgrade. Daemon helpers live in `internal/daemon/`; the CLI surface lives in `app/backend/cmd/rk/daemon*.go` (one file per subcommand: `daemon.go` parent, `daemon_start.go`, `daemon_stop.go`, `daemon_restart.go`, `daemon_status.go`, plus `daemon_portowner.go` for the shared port-owner helper).

- `rk daemon start [--force]` — start daemon (errors if already running)
- `rk daemon stop [--force]` — graceful shutdown via C-c to the tmux pane; prints `rk daemon not running` and exits 0 when no session exists
- `rk daemon restart [--force]` — idempotent: stop existing daemon (C-c → wait up to `stopGracePeriod` 12s for graceful exit, an independent grace-deadline timer sized to cover the inner serve's ~10s worst-case shutdown — see `internal/daemon` row; a session that vanished on its own counts as a clean stop) then start new one; starts fresh if no daemon running
- `rk daemon status [--json]` — read-only report of daemon state (running / not running, socket, session, window, target) AND port owner (free / held by the daemon / held by another process). `--json` emits a structured machine-readable form (`daemon.{running,socket,session,window,target,pid}` + `port.{host,port,state,holder_pid,holder_command}`)
- `rk update` — auto-restarts daemon after successful `brew upgrade` using `RestartWithBinary(brewBinPath)` where `brewBinPath` is `<brewPrefix>/bin/run-kit` derived from the resolved `/Cellar/run-kit/` path (resolves to the new Cellar version post-upgrade, avoiding stale `os.Executable()` path from the running process; the `run-kit` bin/Cellar identity since `260709-gidk-swap-canonical-cli-name-run-kit`). Operates at the Go API layer, not the CLI surface

The pre-`260528-5zvv` flags `rk serve -d`, `rk serve --restart`, and `rk serve --stop` were removed in a hard break — no deprecation forwarders. Invoking them now surfaces cobra's standard `unknown flag` error. `rk serve` is foreground-only.

**Naming collision note**: `rk daemon status` is distinct from `rk status` (the top-level tmux session summary in `cmd/rk/status.go`). The collision is sibling-namespaced — cobra disambiguates cleanly — but `rk daemon status --help` includes a one-line note pointing this out so users typing `rk status` expecting daemon info land at the right command.

**`--force` semantics** — consistently "override the safety check that prevents claiming the port from a non-rk-daemon owner":

- `rk daemon start --force` — lazy / on-error. The happy path is identical to the non-forced path (`daemon.Start()` → port-probe). Only when `Start()` returns the port-in-use error does the command look up the port owner, refuse to kill if the holder identifies as the rk daemon (`daemon.InnerServePID()` match), otherwise SIGTERM the owner with graceful-then-forceful escalation, and retry `Start()`.
- `rk daemon stop --force` — stop the daemon first (if running), then regardless of step 1's outcome probe the port. If still held by a non-daemon process, SIGTERM it. If the port is free at probe time, exit 0 silently (no `Killed port owner` line). If the holder identifies as the rk daemon itself, exit non-zero with a "manual investigation needed" message.
- `rk daemon restart --force` — stop the daemon if running, then proactively probe the port BEFORE the start step. Any non-daemon holder is SIGTERMed (with graceful-then-forceful escalation) before `daemon.Start()` is called.

`rk daemon status` is read-only and does NOT accept `--force` — passing it produces cobra's `unknown flag` error.

Binary resolution: `Start()` uses `os.Executable()` + `filepath.EvalSymlinks()` (works for the standard `rk daemon start` / `restart` paths where the running binary is valid). `StartWithBinary(path)` accepts an explicit binary path and resolves its symlinks (used by `update` after `brew upgrade` deletes the old Cellar directory). Both delegate to shared `startSession(exe)` for tmux session creation.

Detection: `tmux -L rk-daemon has-session -t =rk-daemon` (anchored `=` prefix forces exact-match — see PR #196). `runningSessionCtx` consults `=rk-daemon` first, then falls back to `=rk` for transparent stop/restart of pre-rename daemons. No polling loop, no signal files, no supervisor script.

**Port-based liveness probe**: Before `Start()`/`StartWithBinary()` create the tmux session — and AFTER the `IsRunning()` check passes — they call `guardPortAvailable()`, which loads `config.Load()` and runs `net.DialTimeout("tcp", host:port, 400ms)` against `RK_HOST:RK_PORT`. Wildcard/empty hosts (`""`, `"0.0.0.0"`, `"::"`, `"[::]"`, `"[::1]"`) substitute `127.0.0.1` via `probeHost()` — dialing `0.0.0.0` itself is platform-inconsistent and not a reliable liveness signal. If the dial succeeds (something is listening), `Start` returns an error containing the substrings `already serving on`, `not under the rk-daemon`, and `RK_PORT` so scripts can pattern-match. Cobra `RunE` propagates the non-zero exit. This closes the split-brain detection gap where a foreground `rk serve` was invisible to the tmux-only `IsRunning()` check.

**Stale-socket reaping**: When `IsRunning()` is false AND the port probe passes, `Start`/`StartWithBinary` call `reapStaleDaemonSocket(ctx)` which invokes `runTmux(ctx, "kill-server")` — scoped strictly to `serverSocket = "rk-daemon"` (never the agent-session `runkit` server). Idempotent: tmux's "no server running" surface on cold start is the common case. Errors are logged at `slog.Debug` and never block startup; if anything is genuinely broken, the subsequent `startSession` surfaces it. This self-heals the orphaned-socket state that arose when a previous inner serve died on a port collision before this fix shipped.

**Startup logging via `RK_DAEMON_LOG`**: `startSession` resolves `os.UserCacheDir()/rk/daemon.log` (e.g. `~/.cache/rk/daemon.log` on Linux, `~/Library/Caches/rk/daemon.log` on macOS) via `resolveDaemonLogPath()` and passes it to the inner `tmux new-session` via `-e RK_DAEMON_LOG=<path>`. The serve startup's `setupSlog(level)` reads the env var, `os.MkdirAll(filepath.Dir(p), 0o755)`, opens the file with `O_APPEND|O_CREATE|O_WRONLY` (mode 0o644), and wraps `io.MultiWriter(os.Stderr, file)` so slog output is teed to both the tmux pane (preserves `tmux attach` debugging) and a durable log file. Open failures fall back to stderr-only with one `slog.Warn("daemon log unavailable", ...)`; `UserCacheDir()` failure in `startSession` spawns the inner serve without the env var (single `slog.Warn` upstream). Logging is best-effort — never blocks daemon creation. Constitution §VI compliance: the log is a passive append-only artifact, no supervisor, no signal file, no liveness derivation from contents.

The exported constant `daemon.LogEnvVar = "RK_DAEMON_LOG"` is the single source of truth (consumed by both `internal/daemon/startSession` and `cmd/rk/serve.go/setupSlog`).

**Port-owner lookup (`daemon_portowner.go`)**: shared helper used by all four `rk daemon` subcommands. `findPortOwner(ctx, host, port) (*PortOwner, error)` queries who is listening on the configured TCP port via `lsof -ti:<port>` (primary), falling back to `ss -tlnp '( sport = :<port> )'` when `lsof` is unavailable or errors. Both queries are port-only — the `host` argument is pass-through for display (both tools correctly cover loopback and wildcard binds). Holder command is resolved via `/proc/<pid>/comm` on Linux, `ps -p <pid> -o comm=` on macOS. Returns `(nil, nil)` when nothing is listening; `(nil, error)` only when both tools fail. The companion `terminateOwner(ctx, owner)` mirrors `daemon.Stop`'s graceful escalation: `syscall.Kill(pid, syscall.SIGTERM)` → 200ms polling for exit via `syscall.Kill(pid, 0)` up to a 5s grace period → `syscall.Kill(pid, syscall.SIGKILL)` on timeout. Signal delivery uses `syscall.Kill` directly — never a shell `kill` invocation. All subprocess calls (`lsof`, `ss`, `ps`) use `exec.CommandContext` with a 5s `context.WithTimeout` per Constitution §I. `var findPortOwner = findPortOwnerImpl` is overridable in tests so CLI tests drive `--force` paths without spawning the real tools.

**Holder-identity check (`daemon.InnerServePID()`)**: small helper in `internal/daemon` that returns the PID of the `rk serve` process running inside the daemon tmux pane, derived from tmux's `pane_pid` format spec via `tmux -L rk-daemon list-panes -t =rk-daemon:=serve -F '#{pane_pid}'` (a new `runTmuxOutput` internal captures stdout where `runTmux` would have discarded it). Returns `(0, error)` when the daemon session is absent or the tmux query fails. Used by every `--force` code path and by `rk daemon status` to recognize the daemon as the port owner — when `owner.PID == innerServePID`, the port is classified as held by us (no signal sent in `--force`; `held-by-daemon` state in `status`). The CLI layer wires this through a package-level `var innerServePIDFn = daemon.InnerServePID` so tests can drive the self-recognition branch.

## tmux Control-Mode Subscription (`internal/tmuxctl`)

Package `app/backend/internal/tmuxctl/` owns the long-running `tmux -CC` control-mode subscriptions that drive sub-500ms SSE freshness. It is the **only sanctioned bypass** of the `internal/tmux/` boundary — every other tmux interaction goes through the request-response helpers in `internal/tmux/tmux.go` with their 10s `context.WithTimeout`. The bypass exists because a single long-lived subscription is structurally incompatible with the `exec.CommandContext`-plus-timeout shape of `internal/tmux/`. The package's `doc.go` documents the boundary explicitly.

### Core types

- **`Client`** (`client.go`) — one PTY-backed `tmux -CC` subscription per tmux server (identified by socket name). Runs `tmux [-L <socket>] -CC attach-session -t =<bootstrap> -r` via `creack/pty.Start` (the `-r` flag puts the connection in read-only mode — see Design Decision below). The `=`-anchored target prevents prefix-match collisions, matching the convention established in PR #196. A single goroutine reads lines via `bufio.Scanner` and dispatches them through `ParseLine`. Public surface:
  - `Open(ctx, socket, sink) (*Client, error)` — non-blocking; errors only on initial PTY allocation or process start failure.
  - `Close() error` — idempotent; cancels the Client's context, kills the subprocess, terminates the reconnect loop.
  - `Generation() int64` — atomic per-server counter incremented on every handled notification.
  - `Wait(after int64) <-chan struct{}` — returns a channel closed once generation > after. Single-use; callers re-call after each consumed event.
- **`Supervisor`** (`supervisor.go`) — owns `map[socket]*Client` guarded by `sync.Mutex`, plus an `fsnotify.Watcher` on the watch directory. Resolves the watch dir from `TMUX_TMPDIR` if set, else `/tmp/tmux-<euid>/`; missing directories are created via `os.MkdirAll(dir, 0o700)` and watched as sockets appear. `Start(ctx)` enumerates existing sockets synchronously (so the SSE hub never races an empty Client map for sockets that already exist on disk) and then runs an fsnotify event loop. Re-creating a socket of the same name closes the prior Client before opening a new one. `Stop(ctx)` closes every Client and the watcher; bounded 5s in production. `Get(socket) *Client` is the lookup used by the SSE bridge.
- **`EventSink`** (`event_sink.go`) — consumer interface with 8 callbacks (`OnSessionWindowChanged`, `OnWindowAdd`, `OnWindowClose`, `OnWindowRenamed`, `OnSessionsChanged`, `OnLayoutChange`, `OnConnectionLost`, `OnConnectionEstablished`). All callbacks dispatch from a single goroutine — the Client's read loop. Sink implementations MUST NOT block. The production sink in `api/tmuxctl_bridge.go` (`hubSink`) is active since `260530-v6hm-active-window-event-derivation`: it records per-group active-window state into a per-Client `ActiveWindowTracker` (see § SSE Hub Bridge). The SSE hub still observes *change* via the per-server generation counter (that path is untouched); the sink's tracking is an additive side effect that supplies the Tier-1 active-window signal to the fetch path. `NoOpSink` is exported for tests, PTY-unavailable scenarios, and as the degraded sink a `nil` SinkFactory produces.
- **`ParseLine(line string) Event`** (`parser.go`) — pure function (no I/O, no time dependence). Recognises the six relevant tmux 3.6a notifications (`%session-window-changed`, `%window-add`, `%window-close`, `%window-renamed`, `%sessions-changed`, `%layout-change`) and framing markers (`%begin`/`%end`/`%error`). `%output` and `%unlinked-window-*` are silently dropped. Unknown notification names return `UnknownEvent{Raw}` and log `slog.Debug` exactly once per unique name (surfaces protocol additions without log noise). Malformed input returns `MalformedEvent{Raw}` — never panics.

### Reconnect FSM

When the PTY read loop encounters EOF or any error other than a Close-driven context cancellation, the Client calls `sink.OnConnectionLost()` exactly once, tears down the subprocess (`cmd.Process.Kill()` if still running, `cmd.Wait()` to reap), waits a backoff, then retries `pty.Start`. The backoff sequence is `initialBackoff = 250ms` → `500ms` → `1s` → `2s` → `5s` → `5s` … capped at `maxBackoff = 5 * time.Second`. The backoff resets to `initialBackoff` on the first non-`%begin` event after a successful reconnect (see `client.go` `handleEvent`). Reconnect retries continue indefinitely until `Close()` is called. Test injection points: `dialFn` and `sleepFn` are swappable so the FSM can be exercised with stubbed I/O and a fake clock.

### Anchor session (`_rk-ctl`)

`tmux -CC` requires an attached session to emit notifications. `resolveBootstrap` (in `client.go`) **always** creates a hidden anchor session named `_rk-ctl` via `tmux [-L <socket>] new-session -d -s _rk-ctl` — unconditionally on every observed server, not only when the server is empty (changed by `260602-a1wo-prevent-exit-empty-server-death`; the anchor is now a permanent **session floor** that keeps the server's session count above zero so tmux's `exit-empty` cannot reap it — see § Constitution Alignment §VI below and `tmux-sessions.md` § `_rk-ctl` Anchor). The anchor is tagged with the server-scoped user-option `@rk_ctl_keepalive 1` via `set-option -t =_rk-ctl @rk_ctl_keepalive 1` (constant `tmuxctl.AnchorKeepaliveOption = "@rk_ctl_keepalive"`). The tag is a defensive marker only — v1 has no runtime consumer, but it allows future code to identify the anchor without depending on the literal session name. A "duplicate session" error from `new-session` (concurrent `rk serve` created the anchor first) is treated as benign by `isDuplicateSessionError` and the Client proceeds to attach. **Floor vs. attach target are decoupled**: the anchor is always created (the floor), but the control-mode attach target still prefers the first existing real session (`firstSessionName`, which now skips `_rk-ctl` so the always-present anchor is not picked), falling back to `_rk-ctl` only when no real session exists. The anchor is hidden from user-facing surfaces via `tmux.ControlAnchorSessionName` — see `tmux-sessions.md` § `_rk-ctl` Anchor for the filtering chokepoints, the `exit-empty off` backstop, and the explicit-kill-only lifetime contract.

### SSE Hub Bridge (`api/tmuxctl_bridge.go`)

The `api` package owns the bridge between `tmuxctl.Supervisor` and the SSE hub. Three artefacts:

- **`WindowChangeSubscriber`** (interface defined in `api/sse.go`) — the interface the hub uses to receive per-server change notifications. Methods: `Generation(server string) int64` and `Wait(server string, after int64) <-chan struct{}`. Generation semantics mirror `tmuxctl.Client`: every observed notification increments the counter; `Wait(after)` returns a channel closed once `generation > after`.
- **`supervisorSubscriber`** — the production adapter. Forwards `Generation` and `Wait` calls to `Supervisor.Get(server).Generation()` / `.Wait(after)`. When no Client exists for a server (PTY-unavailable host, socket not yet allocated), `Wait` returns a **never-closing** channel via `neverChan()` so `selectFirst` falls through to the safety-net timer rather than busy-looping `FetchSessions`. Exported via `NewSupervisorSubscriber(sup)`.
- **`hubSink`** — the production EventSink, one per Client, built by the `SinkFactory` returned from `NewHubSinkFactory()` (the `Supervisor` calls the factory per socket, passing that socket's own `ActiveWindowTracker` — see § Active-Window Event Derivation below; `NewHubSink()` was removed by `260530-v6hm`). Beyond the generation-counter path (which the Client bumps itself, before invoking the sink, so the hub still observes *change* via `Wait`), `hubSink` records active-window state into its tracker: `OnSessionWindowChanged(sid, wid)` resolves `sid`→group via the tracker's cached map (O(1), no subprocess) and records `wid` as that group's active window (latest-event-wins; an unresolved `sid` is dropped for tracking, never errors/blocks); `OnSessionsChanged` refreshes the `$sid`→group map; `OnConnectionEstablished` re-seeds the tracker (refresh map + seed Tier 1 from current `#{window_active}` per group). The two re-seed callbacks issue read-only tmux queries that could block, so they are **offloaded to a goroutine** (the read loop returns immediately — honoring the EventSink "MUST NOT block" contract); the tracker's concurrency-safety makes a slightly-delayed refresh racing an event harmless. The read-only query seams (`listSessionGroups`/`listActiveByGroup`) are injectable struct fields so tests stub them without spawning tmux. The remaining callbacks (`OnWindowAdd`/`Close`/`Renamed`/`OnLayoutChange`/`OnConnectionLost`) stay no-ops.

`Server.SetWindowChangeSubscriber(sub)` materialises the lazy SSE hub (`initSSEHub`) and stores the subscriber. Safe to call before any SSE client connects. `Server.SetActiveWindowProvider(provider)` (since `260530-v6hm`) injects the Tier-1 `ActiveWindowProvider` (the Supervisor) into the production `prodSessionFetcher` so the REST + SSE fetch paths derive `isActiveWindow` from control-mode events; it is a no-op when the fetcher is a test-injected fake, keeping the seam optional.

### Active-Window Event Derivation (`260530-v6hm-active-window-event-derivation`)

The active-window highlight is **event-derived per session group** rather than read from the base session's `#{window_active}` pointer alone. The base pointer goes stale whenever an attachment moves the active window while another group member keeps an independent active-window pointer — historically the per-WebSocket `rk-relay-*` ephemeral, and still `rk riff` / an external `new-window`; reading the base pointer produced the wrong-highlight bug. This whole derivation subsystem (`internal/tmuxctl/*`) was scoped *investigate-only, NOT deletable* by `260602-qn62-move-based-board-pin-sessions` (it is driven by the `_rk-ctl` anchor and serves the SESSIONS highlight independent of boards) and is **left untouched** — note that `260602-qn62` removed the `rk-relay-*` ephemeral, so the relay is no longer one of the stale-pointer sources, but the subsystem's internal comments still reference relays (out of scope for that change). The fix corrects the *read signal* — it does not change any window creator.

**Tracker** (`internal/tmuxctl/active_window.go`) — `ActiveWindowTracker` is a concurrency-safe, in-memory (Constitution §II) store for ONE server, owned by one `Client` (the read-loop goroutine is the sole writer; the fetch path reads). Two `sync.RWMutex`-guarded maps: `byGroup` (group name → active `@wid`, latest-event-wins via `Set`, bulk re-seeded via `SeedGroups`) and `sidGroup` (`$sid` → group, resolved via `ResolveGroup`, refreshed wholesale via `ReplaceSidGroups`). `Get`/`Snapshot` are the read accessors.

**Ownership/wiring** — `NewSupervisor(factory SinkFactory)` takes a `SinkFactory` (a `nil` factory degrades every Client to `NoOpSink` — control-mode observed only via the generation counter, no tracking). `Supervisor.openSocket` allocates a fresh tracker per socket and binds the sink to it (a close-then-reopen gets a new tracker so stale state never leaks across the gap — `OnConnectionEstablished` re-seeds it). `Supervisor.ActiveWindow(server, group) (wid, ok)` is the server-keyed read accessor — a server with no Client, or a group with no tracked entry, returns `("", false)` so the caller falls back to Tier 2. The Supervisor satisfies `sessions.ActiveWindowProvider` directly. Wired in `cmd/rk/serve.go`: `tmuxctl.NewSupervisor(api.NewHubSinkFactory())`, then after `Start` both `SetWindowChangeSubscriber(NewSupervisorSubscriber(supervisor))` and `SetActiveWindowProvider(supervisor)`.

**Two-tier derivation** (`internal/sessions/sessions.go`) — `FetchSessions(ctx, server, provider)` threads an `ActiveWindowProvider` (nil ⇒ Tier-2-only, exactly today's behavior). Per session it queries `provider.ActiveWindow(server, sessionName)` — the user-facing session name IS the group key after `parseSessions` filtering. **Tier 1 (authoritative):** when a tracked `@wid` exists, the pure helper `applyActiveWindow(windows, trackedWid)` marks exactly the matching window active and clears the rest, overriding the parsed `#{window_active}` flag. **Tier 2 (fallback):** no tracked entry ⇒ the base-pointer flag from `parseWindows` stands. Tier 1, once set, is authoritative — the base pointer never overrides it; external clients (iTerm/Ghostty) also emit `%session-window-changed`, so "latest event wins" subsumes them with no override path. `applyActiveWindow` also enforces the single-highlight-per-session invariant and falls back to Tier 2 for a session when the tracked `@wid` matches no live window (e.g. closed between event and fetch) rather than highlighting nothing. The fetcher reaches `FetchSessions` via `prodSessionFetcher` (holding the provider) in `api/router.go`. Frontend auto-follow needs no change — the existing `app.tsx` URL-writeback effect navigates once the corrected `isActiveWindow` arrives via SSE; board panes pin explicit window IDs and never read it.

### Event-Driven SSE Loop

`app/backend/api/sse.go` refactored its per-server snapshot loop from a 2.5s ticker to an event-driven `select` with a 12s safety-net backstop. Three constants govern the cadence:

| Constant | Value | Role |
|----------|-------|------|
| `safetyPollInterval` | `12 * time.Second` | Backstop when a `WindowChangeSubscriber` is wired (production). Heals missed events across reconnect gaps; primary freshness driver is the Client's `Wait` channel. |
| `legacyPollInterval` | `2500 * time.Millisecond` | Used when no subscriber is wired (PTY-unavailable startup, unit tests). The snapshot-rebuild cadence is then the only freshness guarantee. |
| `metricsPollInterval` | `2500 * time.Millisecond` | Cadence for `metrics.Collector` host CPU/memory polling. Deliberately decoupled from the SSE intervals so metrics sampling frequency is not coupled to the SSE event/safety cadences. |
| `prStatusPollInterval` | `90 * time.Second` | Cadence for `prstatus.Collector`'s single batched `gh` call (`260610-596o-pr-status-sidebar`). Deliberately slow (~40 calls/hr vs. the 5000/hr authenticated limit) — the SSE hot path reads the cached snapshot, never `gh`, so PR-status freshness is decoupled from the SSE cadence; on-demand `POST /api/pr-status/refresh` covers the "I want it now" case. |

`sseHub.safetyIntervalEffective()` picks `safetyPollInterval` when `subscriber != nil`, else `legacyPollInterval` — preserves correctness for the PTY-unavailable case and for tests exercising the hub without a control-mode driver.

The loop body (`poll` + `waitForNext` + `selectFirst`) blocks on a `select` over (a) one `subscriber.Wait(server, perServerGen[server])` channel per active server and (b) `time.NewTimer(safetyIntervalEffective())`. When a subscriber fires, the winning server's prior generation is refreshed and the server is marked event-driven so the next iteration invalidates its 500ms `cache` entry and observes the post-mutation tmux state immediately. Non-winning servers are peeked non-blockingly so their backlog isn't replayed unnecessarily. `selectFirst` reads `timer.C` in the outer select to avoid the goroutine leak that would otherwise occur if a subscriber wins the race and the timer goroutine blocked on `timer.C` forever (`Stop` does not deliver on `C`). Heartbeat is wall-clock-based (`sseHeartbeatPeriod = 15 * time.Second`) — wall-clock cadence prevents the slower safety interval from starving heartbeats.

### No Killed-Window Board Cleanup Diff (removed)

Since `260602-qn62-move-based-board-pin-sessions`, the SSE hub no longer detects killed windows to clean up board membership. The old `detectKilledWindowIDs(prev, current) []string` pure function, the per-server `sseHub.previousWindowIDs` tracking, the per-tick window-kill cleanup loop, and the `BoardEntriesFetcher.RemoveAllByWindowID` dependency are all **deleted**. There is also no `@rk_board` first-poll bootstrap broadcast (`broadcastBoardBootstrap`/`boardBootstrapPayload`/`previousBoardJSON` removed) and no cached-board send on `addClient`. `BoardEntriesFetcher` is now a one-method interface (`ListBoardEntries`) kept only so tests can stub the tmux dependency; `broadcastBoardChanged` (pin/unpin/reorder) remains and the `"cleanup"`/`"bootstrap"` change strings are gone.

### PR-Status SSE Join (`260610-596o-pr-status-sidebar`)

The SSE hub attaches live PR status to **any window with a derived `PrURL`** on each poll via a PURE in-memory read — the 2.5s hot path makes ZERO network/`gh` calls (the `gh` cost lives entirely on the collector's 90s background tick, the branch→PR refresher's 30s tick, and the on-demand `POST /api/pr-status/refresh`). Since `260705-dmex-generic-agent-state-tier` the change-bound gate is gone: `PrURL` is derived from the pane's branch (§ Branch→PR Derivation), so PR status now appears for any pane on a branch with an open PR, not only fab-change-bound windows.

**Seam + wiring**: `PRStatusSnapshotter` (one method, `Snapshot() map[int]prstatus.PRStatus`, in `api/sse.go`) is implemented by `*prstatus.Collector` and held on the hub (`sseHub.prStatus`). `newSSEHub(fetcher, mc, pc)` takes it as a third arg; `NewRouterAndServer` constructs `pc := prstatus.NewCollector(prStatusPollInterval); pc.Start(ctx)` next to the metrics collector, stores it on `Server.prStatus`, and `Server.initSSEHub` passes it through — but as a true nil interface when `s.prStatus` is nil (the explicit `var pc PRStatusSnapshotter; if s.prStatus != nil { pc = s.prStatus }` wrap avoids the typed-nil-interface trap so the hub's `h.prStatus == nil` check actually fires). A nil `prStatus` degrades gracefully (no PR fields attached) — used by tests and unwired routers.

**`attachPRStatus(sess []sessions.ProjectSession)`**: called in `poll()` AFTER the 500ms session-cache write (so the cached snapshot slice stays free of mutable PR fields conceptually, though it operates on the same slice by reference). For every window it FIRST resets the four display fields (`PrState/PrChecks/PrReview/PrIsDraft` → zero) so a window that lost its PR (merged/closed → dropped from the snapshot) clears cleanly even on a cached result slice; THEN, gating only on a **non-empty `PrURL`** (`w.PrURL == nil || *w.PrURL == ""` → skip; the `FabChange` gate was **dropped** by `260705-dmex-generic-agent-state-tier` per R7), it looks `*w.PrURL` up in the URL-keyed `Snapshot()` and copies `State/Checks/ReviewDecision/IsDraft` onto the window. The join key is the **canonical PR URL, never a bare number** — numbers are unique only per repo, so a number join can pick up an unrelated repo's PR state (the collision the URL-keying fixed). No-op when `h.prStatus == nil`. `PrURL`/`PrNumber` are NOT touched here — since `260705-dmex` they come from the branch→PR enrichment in `internal/sessions` (`enrichWindowPR` → register + snapshot-join against the `prstatus` branch refresher), upstream of the hub.

### Branch→PR Derivation (`260705-dmex-generic-agent-state-tier`)

`PrURL`/`PrNumber` are **derived server-side from the pane's git branch**, no longer joined from `fab pane map` (Constitution §X — PR links are derivable, not pushed). This surfaces PR status for **any** pane on a branch with an open PR, in any repo under any workflow — not only fab-change-bound windows.

**Two layers, both network-isolated from the SSE hot path:**

1. **Background `BranchRefresher`** (`internal/prstatus/prstatus_branch.go`): resolves each registered `(repoDir, branch)` → PR via `gh pr list --head <branch> --state all --json number,url,state,updatedAt` (run in the repo via `cmd.Dir`, `exec.CommandContext` + `ghTimeout` + explicit argv — §I), on a 30s tick (`branchPRRefreshInterval`, mirroring `Collector.Start`'s immediate-warm-then-tick lifecycle). It gates on ONE availability probe per pass (`checkAvailable` — the `ghAvailable` verdict is cached positive AND negative for 60s so an unauthenticated gh never re-probes per pair), keeps the last-good entry on a transient exec/network error OR malformed JSON (true stale-while-revalidate — never fail-to-negative), records a **successfully parsed empty/no-PR result as a valid negative** (`nil pr` — clears the entry directly; no grace/negative-stamp retention), and ages out pairs no longer observed after 5m. `pickBranchPR` selects by **precedence — open > merged > closed** (`branchStateRank`, case-insensitive), most-recently-updated within a class, skipping URL-less nodes and returning `nil` for an empty/all-skipped array. Started in `router.go` via `prstatus.DefaultBranchRefresher.Start(ctx)` next to the viewer-wide collector.

2. **Hot-path register + snapshot-join** (`internal/sessions.enrichWindowPR`): the FetchSessions loop does ZERO network work — it `prstatus.Register(repoDir, branch)`s the observed pair (a cheap lock-guarded set touch) and `prstatus.SnapshotBranchPR(repoDir, branch)`s the last-good derived PR from the in-memory snapshot, assigning `WindowInfo.PrURL`/`PrNumber`. The window's `(repoDir, branch)` is `windowBranchRepo`'s choice: the active pane's cwd/branch, else the first pane with a resolved branch.

**D2 merged durability — state-all + precedence, stateless/restart-proof** (revised by `260706-4h26-durable-merged-pr-register-keys`; status-pyramid.md § Open Decisions — D2): the earlier `260706-y1ar` shape (`--state open` lookup + a `branchPRMergedGrace` 10-min grace clock on a `wentNegativeAt` field) DROPPED a PR the moment it merged, then leaned on an in-memory grace window to keep serving the last-known PR so the purple/orange **done-square** survived. That grace clock **decayed the merged square into a green fab square minutes after merge** and was **wiped by any rk restart** (in-memory state, Constitution II). The revised model derives durability **from `gh` directly, statelessly**: the branch query is now `--state all` (§ prstatus package row + layer 1 above), so a merged PR keeps resolving to a positive result on **every** refresh pass — `pickBranchPR` ranks it by precedence (open > merged > closed) and serves it as the last-good entry. There is **no grace clock to expire and none to wipe on restart** — a cold `BranchRefresher` (fresh process, no prior positive) serves the merged PR on its first pass, so the done-square is durable **statelessly and restart-proof end-to-end**. `refresh` now clears the entry directly on a successfully parsed empty/no-PR result (a true negative) and keeps last-good only on a transient exec error or malformed JSON (stale-while-revalidate) — the `branchPRMergedGrace` const, the `wentNegativeAt` field, and the grace/negative-stamp retention branch in `refresh` were **deleted**. The viewer-wide collector (which queries OPEN/MERGED/CLOSED) supplies the merged `prState` via the URL join, so the dot shows the done-square; a **closed-unmerged** PR (still derived for the register/tip) has its fall-back to the live fab tier handled **frontend-side** (`statusDotState`'s `prOwnsDot` drops PR ownership for `prState === "closed"` — see [ui-patterns](/run-kit/ui-patterns.md) § Status Dot D2). The **branch-reuse edge**: an open PR always outranks an older merged PR on the same branch (state precedence beats recency across classes), so a reopened branch's live PR wins.

This is the T007/T008 **rework shape** (from `260705-dmex`): the original apply resolved `gh` inline in the enrichment loop, which broke `api/sse.go`'s zero-network-call hot-path invariant and code-review.md's 5s cap; review flagged it must-fix and it was restructured to register-off-tick + snapshot-join. The `prstatus` viewer-wide collector (URL-keyed live state/checks/review) is untouched and composes exactly as before: derive `PrURL` here → `attachPRStatus` joins live status by that URL.

The old **`FabChange` gate is dropped** from `attachPRStatus` (R7) — see § PR-Status SSE Join. The cross-repo contract for the fab-kit divestment this enables is in [`docs/specs/agent-state.md`](../../specs/agent-state.md); the run-kit-side agent-state tier that landed alongside it is [agent-state](agent-state.md).

### Web Push on Sustained Waiting (`260706-y1ar-status-pyramid-ui-surfacing`)

The SSE poll seam fires exactly **ONE Web Push per sustained-`waiting` episode** — an agent blocked on a HUMAN (the most notification-worthy state) that stays `waiting` for ≥ `waitingPushSustain` (15s). `idle`/`active` never push. It rides the per-server poll tick where the rolled-up window `AgentState` already exists — **no new goroutine, no new poller** (Constitution II applies to *durable* state; the episode map is in-memory only, mirroring the metrics/prstatus collectors). Logic lives in `api/waiting_push.go`; the seam wiring is in `api/sse.go`'s `poll()`.

**Episode model + tracker** (`waitingPushTracker`, own mutex — independent of the hub lock): a waiting EPISODE is a maximal run of consecutive `waiting` ticks for one window, keyed `waitingKey(server, windowID)` (`server\x00windowID` — window ids are unique only within a server). `waitingEpisode{since, pushed}` stamps when the current run began and whether it has pushed. The **pure `decide(wins []pushWindow) []waitingPush`** advances the tracker for one server's windows at `now` and returns the pushes to send: waiting + no run → start the run; waiting + run < sustain → wait; waiting + run ≥ sustain + `!pushed` → PUSH once + mark pushed; not-waiting → **delete the key** (re-arm — the next `waiting` run is a fresh episode that can push again). It is pure w.r.t. its inputs + the tracker map (the only side effect is mutating that map, which IS the episode memory). `decide` is the extracted testable helper (drives enter/sustain/push/continue/leave/re-enter — A-022/A-023).

**Sync decision, detached send** (T014 rework — the load-bearing invariant): `notifyWaiting(server, sess)` runs `decide` **synchronously** (in-memory map mutation, no I/O — the caller observes the tracker advance in-tick) but fans the actual `push.Notify` sends out in a **detached `context.Background()` goroutine** — a slow/hung Web Push endpoint can NEVER stall the SSE poll loop's documented zero-network hot path. Push errors are logged (`slog.Warn`), never surfaced (matches the `/api/notify` posture). Body: window name + `"waiting for input"`. The `notify` send fn is a seam (defaults to `push.Notify`, stubbed in tests).

**Scoped reap** (`retain(live, polled)` — the other T014 rework): the poll loop accumulates `liveWaitingKeys` across all polled servers, then reaps episodes for windows that vanished (so a re-created window id can't inherit a stale `pushed` flag). The sweep is **SCOPED to servers actually observed this tick** — the ones successfully polled (`polledServers`) PLUS the ones confirmed gone (`deadServers`). A server whose fetch failed TRANSIENTLY (non-`IsServerGone`) is in neither set and contributes zero live keys, so scoping the sweep leaves its still-waiting episodes untouched — otherwise its `since`/`pushed` would reset every failing tick, re-arming and firing a DUPLICATE push the moment the server recovered. `retain` recovers the server from a key by splitting on the NUL separator and reaps only `polled[server] && !live[key]`.

**Wiring**: `sseHub` gained a `waitingPush *waitingPushTracker` field (`newWaitingPushTracker()` in `newSSEHub`); in `poll()`, after `attachPRStatus(result)` for each successfully-fetched server, `h.waitingPush.notifyWaiting(server, result)` runs and its returned live keys accumulate into `liveWaitingKeys`; after the server loop, `h.waitingPush.retain(liveWaitingKeys, polledServers∪deadServers)` sweeps. Uses the existing `internal/push` subsystem (§ Web Push Notifications) — no new endpoint. `@rk_agent_msg` question text in the body is a **future tenant** (out of scope; the body is the fixed `"waiting for input"`).

### Session-Tile Previews — `event: preview` + preview-scope (`260701-70a0-session-window-tile-grid`)

The `SessionTiles` density view (`ui-patterns.md` § Session Tiles) shows a periodic **text snapshot** of each expanded window's active pane. Snapshots ride the existing SSE hub as a **dedicated `event: preview`** — deliberately NOT extending the `sessions` payload and NOT a live xterm/relay per tile (that would re-trigger the documented HTTP/1.1 6-per-origin pool starvation — the load-bearing constraint). This decoupling keeps the `sessions` dedup/order machinery lean (growing preview text never bloats the sessions dedup cache) and lets preview cadence stay independent of structural updates. `WindowInfo` is **unchanged** — no `preview` field.

**Per-connection expanded-set scope**: each `sseClient` carries `connID` (from the `?conn=` stream query param) and `expanded map[string]bool` (the session names that connection currently has expanded). Both are in-memory only (Constitution II) and drop with the client on disconnect (`removeClient` deletes the whole `*sseClient` — no leaked hub state, no separate cleanup path). `setPreviewScope(server, connID, expanded)` finds the client whose `connID` matches (under the hub lock) and replaces its `expanded` set; a `connID` miss is a silent no-op. It also immediately emits the cached preview subset for the just-changed set via `sendCachedPreviewLocked` (a no-op when the set is empty or nothing is cached).

**Capture in `poll()` (rides the existing tick — no new goroutine/loop)**: after the `sessions` broadcast for a server, the poll computes `expandedUnionLocked(clients)` — the union of every client's expanded session names on that server. When the union is empty (capture-nothing default — previews are opt-in per expansion), nothing is captured. Otherwise `capturePreviews(result, union, server, h.captureFn)` captures each window in a union session **once per tick, deduped by window ID** (`h.captureFn` defaults to `capturePreviewForWindow`; tests inject a stub so the broadcast path runs without a live tmux server). The full `{windowId → text}` union map is cached in `previousPreviewJSON[server]` (seeds cached-on-connect), then each client is sent a **per-connection-filtered subset** (`previewSubsetFor`, keyed off its own `expanded` set via `windowsBySession(result)`) over `event: preview`. The tmux captures run OUTSIDE the hub lock (each is an exec with its own timeout); only the cache write + per-client emit take the lock.

**Capture primitive (reused, no new tmux helper)**: `capturePreviewForWindow(w, server)` in `api/preview.go` picks the window's active pane via `activePaneID(w)` — the `PaneInfo` in `WindowInfo.Panes` whose `IsActive` is true, falling back to the first pane, `("", false)` when the window has no panes — then calls the existing `tmux.CapturePane(paneID, previewCaptureLines, server)` (Constitution I — `exec.CommandContext` + timeout, arg slices). `previewCaptureLines = 18` (named constant, within the ~15–20 range). Best-effort: a no-pane window or a capture error yields no entry (skipped, never a hard failure). **No new tmux resolution helper was added** — the active pane ID is already on `WindowInfo.Panes` from `ListWindows`'s `list-panes` enrichment (apply Assumption #1; the intake's conditional `tmux-sessions` memory modify did NOT trigger).

**Cached-on-connect**: `addClient` calls `sendCachedPreviewLocked` after the metrics cached-on-connect send, mirroring the `sessions`/`session-order`/`metrics` pattern — filtered to the new client's current expanded set (a no-op for a fresh connection, since scope is declared by a POST that arrives after connect).

**Frontend seam** (`app/frontend/src/contexts/session-context.tsx`): `SessionProvider` generates a per-pooled-server `connId` (`crypto.randomUUID()`), appends `&conn=<id>` to the stream URL, mirrors it into `connIdByServerRef`, listens for `event: preview` (merging the `{windowId → text}` map into that server's slice so a preview for one window doesn't clobber the others), and exposes `previewsByServer: Map<string, Record<string,string>>` + a `setPreviewScope(server, expanded[])` context method. That method looks up the server's `connId` and POSTs `/api/preview-scope` via the `setPreviewScope(server, conn, expanded)` client fn in `app/frontend/src/api/client.ts` (imported as `apiSetPreviewScope`); errors are swallowed — a preview is a nicety. The `SessionTiles` view calls the context method whenever its expanded set changes.

### `rk serve` Wiring

`cmd/rk/serve.go` instantiates the Supervisor after `tmux.EnsureConfig()` but BEFORE the `server.ListenAndServe()` goroutine, so the SSE hub never races an empty Client map for sockets that already exist on disk. (Since `260602-qn62-move-based-board-pin-sessions` there is no longer a `sweepOrphanedRelaySessions(ctx)` step between them — relay ephemerals are gone and pin-sessions are persistent, so `serve_sweep.go` was deleted and the startup ordering dropped to `EnsureConfig → Supervisor.Start → ListenAndServe`.) The Supervisor is constructed with `api.NewHubSinkFactory()` (since `260530-v6hm` — the per-socket tracker-bound sink factory; was `api.NewHubSink()`) and started synchronously; per-socket `Open` failures (PTY unavailable, etc.) are logged inside the Supervisor and never block startup. Once `Start` returns, `apiServer.SetWindowChangeSubscriber(api.NewSupervisorSubscriber(supervisor))` wires the change-notification bridge and `apiServer.SetActiveWindowProvider(supervisor)` wires the Tier-1 active-window read path (see § Active-Window Event Derivation) — if `Start` itself fails (rare; usually fsnotify allocation), a single `slog.Warn` falls back to safety-net-only mode without aborting the server.

On SIGINT/SIGTERM, the shutdown handler runs `supervisor.Stop(ctx5s)` BEFORE `server.Shutdown(ctx)`. Stop errors are logged at `slog.Warn` and never block shutdown — matches PR #197's daemon-log graceful-degradation pattern. The 5s bounded teardown matches the existing `tmux.TmuxTimeout = 10s` shape (control-mode subprocesses are expected to close cleanly within half that bound).

### API Server version/checker wiring (`260713-4zap-update-notify-one-click-upgrade`)

The version-exposure + update-checker wiring lives in `cmd/rk/serve.go` (NOT `NewRouterAndServer`) because it needs the ldflags-injected `main.version`, which is known only in `cmd/rk`. Immediately after `router, apiServer := api.NewRouterAndServer(ctx, logger)`:

```go
apiServer.SetVersion(version)                                       // seed the SSE event: version slot + hold on Server
updateChecker := updatecheck.New(version)                          // construct with the running version
updateChecker.OnQualify = apiServer.WireUpdateAvailableBroadcast() // bind BEFORE Start
updateChecker.Start(ctx)                                           // background poll, bound to serve ctx
apiServer.SetUpdateChecker(updateChecker)                          // inject so /api/update reads its snapshot
```

The three `*api.Server` seams (all in `api/tmuxctl_bridge.go`, co-located with `SetWindowChangeSubscriber`/`SetActiveWindowProvider`):

- **`SetVersion(version string)`** — stores `s.version = version` (a held copy) AND calls `s.initSSEHub()` + `s.sseHub.setVersion(version)` to seed the cached `event: version` slot. Safe to call before any SSE client connects. **Latent dead field**: the functional version flows via the `version` *argument* into the SSE slot — `s.version` (the struct field, `router.go`) is written here but read nowhere, so it is currently **write-only** (flagged as a review should-fix on this change's *new* code, NOT a pre-existing deletion candidate; the doc comment anticipates a future reader).
- **`SetUpdateChecker(c *updatecheck.Checker)`** — stores `s.updateChecker = c` so `handleUpdate` reads its `Snapshot().Qualifies`. A nil checker makes `/api/update` report `409 no update available` for every request (the test/unwired-router degradation).
- **`WireUpdateAvailableBroadcast() func(current, latest string)`** — `initSSEHub()`s then returns a closure `func(current, latest) { s.sseHub.broadcastUpdateAvailable(current, latest) }`, assigned to `checker.OnQualify`. The end-to-end chain: `updatecheck.checkOnce` → `OnQualify(current, latest)` → this closure → `broadcastUpdateAvailable` → caches `cachedUpdateAvailableJSON` + fans `event: update-available` to all clients.

The whole block sits before the tmuxctl Supervisor start and the HTTP listen. The checker suppresses itself for the `"dev"` sentinel / unparseable versions (so a local `go build` never fetches or notifies), and its goroutine exits on `ctx.Done()`.

### Constitution Alignment

- **§I (Security First)** — all subprocess calls in `tmuxctl/` use `exec.CommandContext` with explicit argument slices; no shell strings. The Client's `tmux -CC` invocation has no overall timeout because the connection is long-lived (per-call timeouts are not appropriate); the context is cancelled by `Close()` and propagates through the goroutine.
- **§II (No Database)** — no file is written under `~/.run-kit/`, `~/.cache/rk/`, or anywhere else. The Supervisor's `map[socket]*Client` mirrors the kernel-observable filesystem view of tmux sockets; nothing persists across `rk serve` lifetimes. The only persistent artefact in this neighbourhood is `~/.cache/rk/daemon.log` from PR #197, unchanged by this package.
- **§VI (Tmux Sessions Survive Server Restarts)** — `tmuxctl` observes tmux state changes and pushes them to SSE clients; it never mutates *user* sessions (it creates/reads only the `_rk-ctl` anchor and sets the server-scoped `exit-empty off` option — never touching any window, pane, or user session). Since `260602-a1wo-prevent-exit-empty-server-death` the `_rk-ctl` anchor is created **on every observed server** (including the user's `default` with live sessions), as a permanent session floor — NOT, as before, only on servers with zero pre-existing sessions. This is itself the deliberate Constitution VI *prevention*, not a violation: the floor + the server-global `set-option -g exit-empty off` (set imperatively in `productionDial` before the anchor on every dial/reconnect, plus baked into the embedded conf) together stop tmux's default `exit-empty on` from reaping a server when its session count momentarily hits zero — the recurring failure (`runWork`/`utils`/`kit`, ≥3x) that killed live agent sessions. (Pre-`260602-qn62` this happened when the last real session closed and the remaining `rk-relay-*` ephemerals drained to zero; since `qn62` removed those ephemerals, the same backstop now also keeps an *empty home session* alive after its only window is pinned away into a `_rk-pin-*` session — pin-session persistence relies on `exit-empty off`.) The anchor is harmless on a server with real sessions: it is filtered from every user-facing list (`tmux.ControlAnchorSessionName` chokepoint in `parseSessions`), ungrouped, and the control-mode connection attaches read-only (`-r`). Managed servers now die only via explicit `kill-server` / `rk reaper`; empty anchor-only servers persist by design (no auto-reaping, no cross-process state per §II).
- **No new network surface** — the Client is a local subprocess attached via PTY. `ss -tlnp` still shows only the existing HTTP listener.

```
grid-template-columns: 0 1fr                   (when !sidebarOpen)
transition: grid-template-columns 150ms ease-out
```

Sidebar is full-height — it spans all three rows and owns the entire left column. Topbar/content/bottombar stack in the right column. Sidebar collapse zeroes the first grid column (no 48px rail) — the topbar/content/bottombar cells extend their left edge to page.left automatically; no JS-side conditional rendering or component mutation. The drag-resize handle is hidden when `sidebarOpen === false` (the only re-open affordance is the hamburger). Drag-handle wiring is AppShell-only — `BoardPage` does not need a drag-handle (the dimension lives in `ChromeContext` and is shared).

Mobile topology (`< 640px`): single column, `grid-template-areas: "topbar" / "content" / "bottombar"`. Sidebar renders as an overlay child of the Shell positioned via `gridRow: "2/4"` (spans content + bottombar, leaves the topbar visible). Backdrop and `<aside>` use `position: absolute` (NOT `fixed inset-0`) so the overlay starts below the topbar — matches the existing project convention recorded in `fab/project/context.md`. The aside carries `role="dialog" aria-modal="true"`. Sidebar overlay open/close is driven by the same `sidebarOpen` state — `drawerOpen` is removed from `ChromeContext`.

**Hamburger placement** — the hamburger icon is statically rendered at `TopBar.left` in both `sidebarOpen` states. Visual "relocation" on collapse is a side effect of the grid column collapsing; there is no component-level relocation logic. When `sidebarOpen === true`, the hamburger sits at the sidebar.right / topbar.left seam (pixel-identical to "attached to sidebar"). When `sidebarOpen === false`, it sits at page.left because the topbar grid cell has extended to page.left.

**Keyboard toggle** — `Cmd+\` (macOS) / `Ctrl+\` (Linux/Windows) toggles `sidebarOpen` from any route that mounts `<Shell>`. The chord is registered in the `KeyboardShortcuts` modal under "Toggle sidebar" per Constitution V (Keyboard-First). The chord is suppressed when focus is inside a real text input/textarea/contenteditable, but NOT when focus is on xterm.js's helper textarea (recognized by `target.closest('.xterm') != null`) — without the xterm exception, the most common focus state would silently swallow the toggle.

**Sidebar width persistence** — width state lives in `ChromeContext` (`sidebarWidth`). The dispatch surface is split: `setSidebarWidth(width)` updates in-memory state only (called per-pointermove during drag), and `persistSidebarWidth(width)` writes `localStorage["runkit-sidebar-width"]` (called once at drag-end). This explicit split avoids localStorage thrash on every pointer event. Callers like the AppShell drag handler track the last-seen width in a ref (`dragLastWidthRef`) and call `persistSidebarWidth` once in `handleEnd`.

**Routing branches inside the `content` grid area**:
- AppShell (`/$server/$window`): `<TerminalClient>` (terminal window) | `<IframeWindow>` (iframe window) | `<Dashboard>` (no window param). The iframe-banner row above `TerminalClient` (when `currentWindow.rkUrl` exists) remains inside `content`.
- BoardPage (`/board/$name`): `DesktopRow` / `MobileCarousel` body (the existing horizontal-scroll panes), flush with `sidebar.right` (no left gutter).

No `max-w-4xl` constraint — all zones span full width.

**ThemeProvider** (`app/frontend/src/contexts/theme-context.tsx`) — outermost provider. Split into ThemeStateContext (preference + resolved theme) and ThemeActionsContext (setTheme). Three modes: system, light, dark. Preference persisted to `localStorage` key `runkit-theme`. System mode listens to `matchMedia("(prefers-color-scheme: dark)")` change events. Theme applied via `data-theme` attribute on `<html>`. Blocking inline script in `index.html` `<head>` reads localStorage and sets `data-theme` before first paint (no-flicker initialization).

**ChromeProvider** (`app/frontend/src/contexts/chrome-context.tsx`) — split into two React contexts: `ChromeStateContext` (read-only state: `currentSession`, `currentWindow`, `sidebarOpen`, `sidebarWidth`, `isConnected`, `fixedWidth`) and `ChromeDispatchContext` (stable setter functions: `setCurrentSession`, `setCurrentWindow`, `setSidebarOpen` — persists to `localStorage["runkit-sidebar-open"]`, `setSidebarWidth` — in-memory only, `persistSidebarWidth(width)` — writes localStorage on drag-end, `setIsConnected`, `toggleFixedWidth`). `drawerOpen`/`setDrawerOpen` were removed when the rotated shell collapsed the desktop-vs-mobile sidebar split into a single `sidebarOpen` boolean (the rendering form differs per breakpoint, the state does not). Chrome derives its content from the current selection — no slot injection. `setLine2Left`, `setLine2Right`, `setBottomBar` removed; top bar and bottom bar read the selection directly. `Breadcrumb` type includes optional `dropdownItems` for session/window switching.

**FocusedTerminalContext** (`app/frontend/src/contexts/focused-terminal-context.tsx`) — tracks the currently-focused terminal across the app so the shell-level `BottomBar` can target the right WebSocket regardless of route. Mounted in `RootWrapper` between `SessionProvider` and `OptimisticProvider`; provider tree order is `ThemeProvider > ChromeProvider > RootWrapper(SessionProvider > OptimisticProvider > FocusedTerminalProvider > Outlet)`. Shape:

```ts
type FocusedTerminal = {
  wsRef: React.RefObject<WebSocket | null>;
  server: string;
  session: string;
  windowIndex: string;
} | null;

type FocusedTerminalContextValue = {
  focused: FocusedTerminal;
  setFocused: (t: FocusedTerminal) => void;
  composeOpen: boolean;
  setComposeOpen: (open: boolean) => void;
};
```

Producers: `TerminalClient` registers on mount when its `registerFocus` prop is `true` (single-terminal route default) and clears on unmount. `BoardPane` registers when its `isFocused` prop becomes true (and on initial pane mount); it manages the registration itself and passes `registerFocus={false}` to its inner `TerminalClient` to avoid producer contention. `BoardPane` does NOT clear on focus loss — the next pane to gain focus overwrites.

Consumers: the shell-level `BottomBar` reads `focused?.wsRef` (replacing the removed `wsRef` prop); the focused pane's `TerminalClient` reads `composeOpen` and gates `<ComposeBuffer>` rendering on `isFocused && composeOpen` (board route) or `composeOpen` (single-terminal route). `composeOpen` persists across route navigation by design — the user dismisses explicitly via Escape. The compose target is frozen at open-time via `compose-buffer.tsx`'s existing wsRef snapshot on mount; cycling focus while compose is open does NOT retarget. When `focused === null` (e.g., Dashboard route with no terminal), BottomBar input handlers natural-no-op via the existing `wsRef.current?.readyState !== OPEN` guard.

**SessionProvider** (`app/frontend/src/contexts/session-context.tsx`) — root-mounted multi-server React Context. One `SessionProvider` instance lives above `<Outlet />` in `RootWrapper` and is shared by every route (`/`, `/$server/...`, `/board/$name`); it does NOT remount on server-route switches. State is keyed per server: `sessionsByServer: Map<string, ProjectSession[]>`, `sessionOrderByServer: Map<string, string[]>`, `isConnectedByServer: Map<string, boolean>`, `metricsByServer: Map<string, MetricsSnapshot | null>`. It also exposes a scalar `hostMetricsConnected: boolean` (host-metrics source health for the Cockpit connection dot, `260704-9o7k` — see § Server-independent host-metrics subscription). `currentServer: string | null` is dispatched from the matched route via `useMatches()` — `params.server` for `/$server/...`, `null` for `/board/$name` and `/`. Consumers read the per-server slice they need via `ctx.sessionsByServer.get(name)`; there is no transitional accessor — all consumers consume the keyed shape directly. Provider tree order: `ThemeProvider > ChromeProvider > RootWrapper(SessionProvider > OptimisticProvider > Outlet)`. The provider mirrors `currentServer`'s connection state into `ChromeProvider` so existing chrome consumers (top-bar connection dot) keep working unchanged. Persists most-recently active server to `localStorage` key `runkit-server` (single value, not split per server).

**Per-server EventSource pool** — the provider owns one `EventSource` per attached server, all connecting to `/api/sessions/stream?server=<name>`. The current server is auto-attached; non-current servers attach lazily via the `attachServer(name)` API exposed on context. Each connection updates only its own slice in the per-server maps. When a server disappears from `/api/servers`, its `EventSource` is closed and its entries are removed from every per-server map. A 3-second disconnect debounce runs per server — an `onerror` on one server does not flip another server's `isConnectedByServer` value. In practice today, `useBoards` and `useWindowPins` call `attachServer` for every known server when the sidebar mounts, so the lazy-attach guard mainly bounds open connections in routes where neither hook is mounted.

**`subscribeBoardChange`** (`session-context.tsx`) — context API that lets any consumer register a `(server: string) => void` handler and receive cross-server `board-changed` SSE events through the existing pool. `useBoards` and `useWindowPins` use it to react to per-server board mutations without opening their own `EventSource` instances — the multi-server pool already streams `board-changed`, so multiplexing through one subscriber API keeps the connection count bounded.

**Server-independent host-metrics subscription + `useHostMetrics()`** (`session-context.tsx`, since `260701-vshd-cockpit-host-console-home`) — a dedicated `EventSource("/api/sessions/stream?metrics=1")` that receives ONLY the server-independent `event: metrics` broadcast, so host health is available where there is no `currentServer` and no per-server stream to carry it (the Cockpit host-console home `/`; see ui-patterns.md § HOST HEALTH Zone). It is **mutually exclusive with the per-server pool**, NOT a permanent extra connection: a `hostMetricsESRef` guard opens it only when `attachedSet.size === 0` (`hostMetricsWanted`), and **closes it once any server attaches** — because the per-server `metrics` listener already fans the same server-global payload into the shared `hostMetrics` state, making the dedicated stream redundant whenever ≥1 server is streaming. This keeps host metrics off the browser's HTTP/1.1 6-per-origin budget on the connection-starvation-fragile board route (which attaches every known server; see ui-patterns.md § Desktop Relay-Connection Suspension). The effect body — not effect teardown — drives the open/close (a cleanup `close()` would orphan the ref-guarded reopen on every StrictMode remount); the ref survives the dev cleanup→re-run cycle so a second run reuses the existing connection. The parsed snapshot lands in provider state `hostMetrics` (`MetricsSnapshot | null`, `null` pre-tick) exposed via `HostMetricsContext` + the `useHostMetrics(): MetricsSnapshot | null` hook. **`useHostMetrics()` is NOT `currentServer`-gated** — unlike `useMetrics()` (which returns `currentServer ? metricsByServer.get(currentServer) : null`, hence `null` on `/`); both hooks use the same undefined-sentinel throw-outside-provider idiom. `useMetrics()` (current-server-scoped, for `HostPanel`'s sidebar call site) is unchanged. Backend contract for the `?metrics=1` sentinel: see § API Layer `/api/sessions/stream`.

**Host-metrics stream health (`hostMetricsConnected`, `260704-9o7k-top-bar-button-pyramid`)** — the health of the host-metrics *source*, exposed on `SessionContextType` as a `hostMetricsConnected: boolean` for the **Cockpit connection dot** (`server-list-page.tsx` passes it as `isConnected` to `<TopBar mode="cockpit">`, replacing the old hardcoded `isConnected={false}`; see ui-patterns.md § Chrome (Top Bar) → connection dot). This closes the gap where the dedicated `?metrics=1` stream's health was previously **untracked**. Two sources, mirroring the dedicated-stream ↔ per-server-fan-out switch above:
- **Dedicated-stream health** — while `hostMetricsWanted` (attached set empty, so the dedicated stream IS the source), a `dedicatedMetricsConnected` state tracks it: set `true` on the dedicated stream's first (or recovered) metrics event, cleared via a **3s `disconnectTimer` debounce** on `onerror` — the same debounce pattern the per-server pool uses, so a transient socket blip doesn't flicker the dot. The timer is held in `dedicatedDisconnectTimerRef` and cleared on the next event, on close, and when a server attaches (the `hostMetricsWanted → false` branch closes the dedicated stream, clears the timer, and resets the flag to `false`). `false` until the first dedicated event.
- **Per-server fan-out fallback** — when a per-server stream is attached (dedicated stream closed, the fan-out now carries host metrics), the derived value reads `true` iff **any** attached server slice `isConnected` (that server is delivering the server-global `event: metrics`).

The exposed `hostMetricsConnected` is a `useMemo` selecting between the two: `hostMetricsWanted ? dedicatedMetricsConnected : (any slice connected)`. It is added to the context type, both provider `value` memos (+ deps), and `StandaloneSessionContextProvider` defaults (`false`).

**Kill-redirect gate (`currentWindowEverSeen`)** — The "session/window gone" redirect in `app/frontend/src/app.tsx` (around the `useEffect` that calls `computeKillRedirect` from `app/frontend/src/lib/navigation.ts`) MUST NOT fire for a freshly-navigated URL whose (session, window) pair has not yet been observed as valid in the SSE `sessions` stream. `computeKillRedirect` accepts a `currentWindowEverSeen?: boolean` param (defaults to `true` for legacy callers and unit tests) and short-circuits to `null` when false — bypassing both the "session gone → dashboard" and "window gone → nearest sibling" branches. The caller maintains two refs: `currentWindowEverSeenRef` (boolean, flipped `true` whenever the memoized `currentWindow` is non-null) and `lastObservedUrlKeyRef` (string, `${server}|${session}|${window}`). Any URL-key change resets the seen flag to `false`. This gates two transient SSE states that are not proof the URL is unreachable: (a) a stale cached first payload that replays an older server snapshot before delivering current data, and (b) a session whose `windows` array is briefly empty while tmux enumeration propagates into the next poll tick. Without the gate, either state makes `currentSession` or `currentWindow` null on the first SSE event after navigation, the redirect fires, and `TerminalClient` unmounts mid-initialization — surfacing as an e2e flake where `.xterm-screen` briefly appears then disappears. Genuine kills from another client still redirect normally, because the URL key only resets on navigation, not on SSE payload changes: once the seen flag is `true`, a subsequent payload that drops the window triggers the redirect as before.

**Window state (Zustand)** — window optimistic state is managed by a Zustand store at `app/frontend/src/store/window-store.ts`, keyed by the immutable tmux `windowId` (`@N`). `zustand` is a production dependency in `app/frontend/package.json`. The store tracks `killed` and `pendingName` overrides per window ID, and ghost windows awaiting SSE confirmation. `AppShell` drives `setWindowsForSession` from SSE `rawSessions`. `OptimisticContext` retains session and server optimistic state only (name-based keys, not index-based). See ui-patterns.md "Zustand Window Store" section for full shape and action surface.

**`WindowInfo` type** (`app/frontend/src/types.ts`) — includes `windowId: string` (non-optional) as the immutable tmux `@N` identifier, alongside `index: number` (tmux ordering, display-only, not used as a store key). Backend populates `windowId` from `#{window_id}` in the `ListWindows()` format string. `color?: string` — optional color-value descriptor (`"4"` single index or `"1+3"` blend) from the tmux `@color` user option (was `number` before `260615-6rnr-expand-swatch-palette-blends`); `ProjectSession.sessionColor?: string` is the analogous per-session field. `panes?: PaneInfo[]` is an optional field (undefined when absent — existing fixtures omit it without type error). `PaneInfo` type: `{ paneId: string; paneIndex: number; cwd: string; command: string; isActive: boolean }`. `rkType?: string` — `"iframe"` or undefined; determines frontend renderer (terminal vs iframe). `rkUrl?: string` — URL for iframe windows; populated from tmux `@rk_url` user-defined option. ## Version Management

- **Source of truth**: `VERSION` file at repo root — plain semver string (e.g., `0.1.0`), no `v` prefix

- **Displayed by**: the `--version`/`-v` global flag (Cobra built-in — the `version` *subcommand* was removed in PR #90; `rk version` exits 1 `unknown command`); also read by `rk update` (shows current version, compares with latest via `brew info --json=v2`)

## Web Push Notifications

`260615-xd9r-web-push-notifications` lets any process on the box — primarily the fab-kit operator agent via `rk notify` — deliver a real OS-level notification that reaches the user **even when the RunKit PWA tab is closed**. This is the only browser-native mechanism that delivers to a closed tab: the *browser's* push service wakes the registered service worker, which shows the notification. It is the intended eventual replacement for fab-kit's `ntfy.sh` default (one feed across all operators on the box, content stays on infra the user runs).

**End-to-end delivery model**: a shell process runs `rk notify "msg"` → POSTs `{title, body}` to the local server `POST /api/notify` → the server iterates stored `PushSubscription`s and sends each a Web Push via `webpush-go` signed with the server's VAPID **private** key → the browser's push service wakes the registered service worker → `sw.js`'s `push` handler calls `self.registration.showNotification(title, {body, icon})` → the user sees an OS notification with the tab closed.

**Service worker** (`app/frontend/public/sw.js`): a hand-written service worker served at the origin root `/sw.js` (whole-app scope). Its `push` handler parses the JSON payload `{title, body, icon?}` (default title `"RunKit"`, default icon `/generated-icons/icon-192.png`) and tolerates an empty / non-JSON payload by falling back to defaults; its `notificationclick` handler focuses an existing RunKit client if one is open, else opens a new window at `/`. **This is a separate file from any Workbox/`vite-plugin-pwa`-generated SW** — the current frontend ships only this hand-written `public/sw.js` and a static `public/manifest.json` linked from `index.html` (the `## PWA Layer` section above describes the earlier `vite-plugin-pwa` build path; the manifest/SW generation tooling is not wired in the current tree). Registered on app load via `navigator.serviceWorker.register('/sw.js')` from `app/frontend/src/main.tsx`, guarded by `'serviceWorker' in navigator` so unsupported / insecure-context environments silently skip registration.

**Secure-context constraint**: service workers and `PushManager` require a **secure context — HTTPS or `localhost`**. The user typically hits run-kit on `localhost:3000` (or behind a TLS reverse proxy such as Tailscale Serve), both secure contexts, so it is satisfied in practice — but raw-IP / plaintext non-localhost access cannot subscribe (registration is skipped silently, no error).

**State persistence** (Constitution §II — No Database): the `internal/push` package owns two JSON files under `~/.rk/` — `vapid.json` (the VAPID keypair, private key written mode **`0600`**, never served) and `push-subscriptions.json` (the subscription array, de-duped by `endpoint`). See § Backend Libraries. There is no `/api/push/unsubscribe` in v1; dead subscriptions are pruned server-side on `404`/`410` during send.

**Fail-silent discipline** (end-to-end, constitution-aligned): every layer fails silently on its own prerequisite being absent. `rk notify` exits 0 with no output on any failure (server unreachable / non-2xx / timeout) so a notify failure never stalls the operator loop; the send fan-out tolerates individual subscription failures and prunes dead endpoints; the frontend subscribe flow aborts silently when permission is denied or the context is insecure / unsupported. The opt-in is a Cmd+K palette action with a terminal-themed label (NOT a bell) — see `ui-patterns.md` § Notifications (Web Push opt-in).

After removal, the `release` job's final step is **Update Homebrew tap**. (Historical: the push step was added in `260602-a36m-help-dump-shll-ai` and hardened in `260602-2dt9-fix-shllai-help-publish`.)

**Homebrew tap update** (now the **final** step in the `release` job): computes SHA256 for all 4 tarballs, clones `sahil87/homebrew-tap` via the `HOMEBREW_TAP_TOKEN` secret (exported as `TAP_TOKEN`), generates `Formula/run-kit.rb` from `.github/formula-template.rb` (placeholder substitution via `sed`), commits with message `run-kit ${version}`, and pushes. ## Boards Feature

Pane boards are named collections of pinned tmux windows rendered as a horizontal pane dashboard at `/board/$name`. Since `260602-qn62-move-based-board-pin-sessions` the model is **move-based and server-scoped**: a pinned window is physically `move-window`'d into its own single-window session `_rk-pin-<id>`, and a board is the **set of pin-sessions sharing an `@rk_board` SESSION var** (not a tmux session itself, and no longer a `@rk_board` server-option encoding). Boards are server-scoped because `move-window` can't cross tmux servers — the board *list* is summarized across reachable servers, but there is no cross-server window union. A board exists only while ≥1 pin carries its name (no empty boards, no registry). See `tmux-sessions.md` § Pin Sessions and § `@rk_board` for the full pin/unpin/reorder contract, the three session vars (`@rk_board`/`@rk_home`/`@rk_board_order`), order keys, and the direct-attach relay.

### Constitution Alignment

- **II (No Database)** — pin state lives only in tmux: the `_rk-pin-*` session structure + its `@rk_board`/`@rk_home`/`@rk_board_order` session vars (a deliberate shift since `260602-qn62` from "a tiny derived server-option value" to "tmux session structure IS the record"). Browser `localStorage` holds only view-local pane widths
- **IV (Minimal Surface Area)** — adds exactly one route (`/board/$name`); `260602-qn62` net-reduced backend surface (deleted the relay ephemeral subsystem, the `@rk_board` encoding, the startup sweep, and the SSE board-cleanup/bootstrap) while keeping one rendering path (`TerminalClient`/`/relay/{windowId}`)
- **V (Keyboard-First)** — pin/unpin/switch/cycle reachable via Cmd+K palette; pane focus cycle via Cmd+[/]
- **VI (Tmux Sessions Survive Server Restarts)** — pin state persists with the tmux server across rk-go restarts, so there is NO restore-sweep (since `260602-qn62`); `tmux kill-server` loses the boards on that server (expected and acceptable). An emptied home session (its only window pinned away) persists via the `exit-empty off` backstop (`260602-a1wo`) so unpin can restore the window to it
- **VII (Convention Over Configuration)** — no new env vars, no new config files; reuses the established `@rk_*` user-option pattern

## Design Decisions

- **Go backend + Vite SPA over Next.js monolith** — decouples frontend and backend for independent iteration. Go backend is a stable, long-lived API that outlives any individual frontend. Multi-client API support (web, mobile, CLI) without split API surface
- **Single port architecture** — Go serves API, WebSocket relay, and SPA static files on one port. The two-port split (Next.js :3000, relay :3001) was a Node.js artifact — separate processes required separate ports. Go serves everything in one binary
- **chi over stdlib ServeMux** — chi for middleware chaining (CORS, logging, recovery). Go 1.22+ ServeMux has pattern matching but lacks ergonomic middleware composition
- **TanStack Router over React Router** — type-safe params and search params, built-in loader pattern. Single route `/:session/:window` in the new frontend
- **Vite proxy in dev (not CORS)** — single browser URL, no CORS config needed. WebSocket upgrade works transparently. Go includes chi CORS middleware for production/non-browser clients
- **SPA fallback in Go** — Go serves standalone; TLS termination handled externally via Tailscale Serve when needed
- **SSE (not WebSocket) for session state** — simpler, server-push only, naturally resilient. Module-level hub deduplicates polling across tabs (one `FetchSessions()` per server per interval). Clients indexed by server name (`map[string][]*sseClient`) — broadcast iterates only the target server's slice. `sync.RWMutex` with `RLock` for read-only poll operations, `Lock` for writes. SSE data includes `isActiveWindow` per window, enabling UI sync when users switch tmux/byobu windows via terminal shortcuts
- **Full snapshots (not diffs)** — small payload (<100 sessions), simple client logic
- **Independent panes per browser client** — no cursor fights, agent pane untouched. The relay pty follows tmux window switches natively (runs `tmux -L runkit attach-session`)
- **Every tmux session is a project** — no config, no "Other" bucket. Project root derived from window 0's `pane_current_path`
- **Config via env vars (not YAML)** — `.env` committed with defaults (`RK_PORT`, `RK_HOST`), `.env.local` for overrides, loaded via `.envrc` (direnv). Go and Vite read `RK_PORT`/`RK_HOST` directly. `dev.sh` translates `RK_PORT+1` for the backend subprocess. No relay port — single port serves everything
- **Dedicated tmux server (`-L runkit`)** — run-kit sessions live on a named tmux server `runkit` with its own config (`internal/tmux/tmux.conf` loaded via `-f`). The default tmux server is also queried for session discovery (read-only display of external sessions). This replaces byobu integration: `CreateSession()` uses plain `tmux new-session` on the runkit server (no byobu dependency). The `runkit` server provides isolation, a custom status bar matching run-kit's dark theme, and byobu-style F-key keybindings (F2/F3/F4). `SessionInfo` carries a `Name` field; `ProjectSession` has `Name` and `Windows` fields. The caller passes the target server name to each tmux operation
- **Multi-server session enumeration** — `ListSessions(ctx, server)` queries the specified tmux server. The API layer calls it once per server (e.g., `runkit` and `default`), merging results. Session-group filtering applies to both servers. See `docs/memory/run-kit/tmux-sessions.md`
- **Derived chrome (not slot injection)** — Single-view model means only one chrome state (terminal-focused). Top bar and bottom bar derive content from the current session:window selection. No `setLine2Left`/`setLine2Right`/`setBottomBar` setters. Split React Context preserved for performance (state vs dispatch).
- **Root-mounted multi-server SessionProvider with per-server EventSource pool** — One `SessionProvider` instance lives at the root above `<Outlet />` (in `RootWrapper`) and is shared by every route. State is keyed per server (`Map<string, T>`); `currentServer: string | null` is dispatched from the matched route via `useMatches()`. The pool opens one `EventSource` per server lazily — current server eager, others via `attachServer(name)`. Provider survives server-route switches (no remount, no state rebuild) and lets the Board route render the same unified `<Sidebar>` as AppShell with `currentServer === null`. Cross-server `board-changed` events are multiplexed through `subscribeBoardChange` so cross-server hooks (`useBoards`, `useWindowPins`) don't open separate connections. Rejected: per-route providers bridged via a sibling cross-server context — adds two contexts, double-fetches `/api/servers`, doesn't solve BoardPage sidebar reuse cleanly. Rejected: `Record<string, T>` instead of `Map<string, T>` — Maps give clear "server unknown" vs "server known with empty value" semantics. (`260508-dc0t-multiserver-session-provider`)
- **Single-view layout (sidebar + terminal/dashboard) replaces three pages** — Dashboard and Project page functionality subsumed by the sidebar + Dashboard view. Terminal is the main content on `/:session/:window`; Dashboard renders on `/`. No page transitions.
- **Dashboard as inline component in app.tsx (not separate route layout)** — The Dashboard renders in the terminal area via a conditional branch (`sessionName && windowIndex ? <TerminalClient/> : <Dashboard/>`). This reuses the existing chrome skeleton (top bar, sidebar) and avoids duplicating the app shell layout. Rejected: separate route with dedicated layout — would require lifting sidebar/chrome into a shared layout component and restructuring the route tree.
- **POST-only API client with path-based intent** — Each mutation is a separate function with its own URL (e.g., `killSession(session)` → `POST /api/sessions/:session/kill`). No multiplexed `action` field in request bodies. Formalized as constitution §IX (Uniform HTTP Verb) and brought into full conformance by `260529-jad6` (the last five `.Put(` routes migrated to `POST`; CORS `AllowedMethods` → `[GET, POST, OPTIONS]`). Semantics that would conventionally map to `PUT`/`PATCH` (partial update) live in the path + body contract instead of the verb.
- **Shared chained-`set-option` primitive + "validate-all-then-one-atomic-invocation" partial-merge** (`260529-jad6`) — `WindowOptionOp{Key; Value *string}` + `appendOptionOps(args, target, ops)` is a single reusable primitive for emitting any sequence of `set-option -w` set/unset ops as one `\;`-chained tmux invocation. Both `SetWindowOptions` (the `POST /options` endpoint) and `CreateWindowWithOptions` (window creation) consume it — there is no separate inline option-map construction path. The `POST /options` handler pairs this with a strict ordering contract: **all keys are validated (allowlist + per-key rules) before any tmux subprocess is spawned**, so a single invalid key aborts with `400` and zero tmux calls (no partial application), and the merge that does run is one atomic chained invocation the SSE poll can never observe half-applied. This validate-all-then-atomic pattern is the reusable shape for future multi-key tmux mutations. *Why*: partial-merge (present key sets, `null` unsets, absent untouched) avoids the "client clears a field it didn't echo" bug class that a full-replace PUT re-introduces; the key allowlist (`@color`/`@rk_url`/`@rk_type`) bounds the option-name surface passed to `tmux set-option` (§I). *Rejected*: a generic action-dispatcher for all window verbs (weaker per-action validation, fuzzier §I audit surface); PATCH/PUT full-replace (violates §IX / re-introduces the cleared-field bug); passing through arbitrary `@`-prefixed keys (unvalidated surface).
- **Session as server-derived disambiguation context for REST `/select`** (`260529-jad6`) — `handleWindowSelect` resolves the owning session server-side (`ResolveWindowSession`) and issues a session-scoped `select-window -t <session>:@N` (`SelectWindowInSession`) rather than a bare `select-window -t @N`. *Why*: a bare window-id target is ambiguous inside a tmux session group (members keep independent active-window state, so tmux may activate the window on the wrong member — the wrong-window-highlight symptom, live since #198 made tmux the sidebar's source of truth). The session is derived only where tmux semantics require it; the client never sends it (consistent with dropping `$session` from the URL). *Rejected*: a bare-target select with a client-supplied session (puts the session back in the client-facing identity).
- **Sidebar + drawer pattern on mobile** — Desktop sidebar is drag-resizable (default 220px, min 160, max 400, localStorage persist), collapsible. Mobile (< 768px) uses a left-side drawer overlay triggered by `☰`. Preserves session/window tree layout across breakpoints.
- **Prebuilt binaries over build-from-source Homebrew formula** — ship cross-compiled binaries via GitHub Release. Zero build dependencies on user's machine (no Go, Node.js, pnpm). Faster install. Rejected: build-from-source formula (like tu) which requires all build tools. (`260317-ukyz-homebrew-deployment`)

- **VERSION file + ldflags over Go constant** — version sourced from `VERSION` file, injected via `-X main.version=...`. Shell scripts can read/write plain text. No code changes for version bumps. Rejected: Go constant (requires code change per release). (`260317-ukyz-homebrew-deployment`)

Full design: [agent-state](agent-state.md), spec [`docs/specs/agent-state.md`](../../specs/agent-state.md). Unblocks fab-kit backlog `[ioku]` (fab pane send/map become convention readers). - **Web update-notify + one-click upgrade: SSE-cached-slot detection + wrap `rk update`, no new deps/routes/DB** — the daemon polls the GitHub Releases API (`internal/updatecheck`, one unauthenticated GET every ~6h) and, on a qualifying minor/major bump, broadcasts a server-global `event: update-available`; the running version rides a set-once `event: version` slot; a `POST /api/update` validates (brew marker + pending update) then spawns a **detached** `rk update`. *Why GitHub Releases API not `brew update`+`brew info` on a schedule*: the brew pair is heavyweight and mutates global brew state every run; a single JSON GET is effectively free and far under the 60/hr/IP unauth limit (user-decided, incl. the rejected alternative). *Why hand-rolled major/minor semver compare*: ~20 lines + a table test, and the repo already hand-parses versions (`parseBrewVersion`) with no semver dep — patch-level never notifies (avoids nagging on bugfix churn). *Why SSE cached slots not a `GET /api/version`*: the two events reuse the exact `server-order`/`board-order` fan-out + cached-on-connect machinery (all clients incl. `?metrics=1`), and the SSE connect IS the moment the client needs the version — a GET adds surface with nothing to poll (§IV). The `version` slot is replay-on-connect-ONLY (the running version can't change mid-process). *Why wrap `rk update` via a detached spawn*: `rk update` already does brew detection + `brew upgrade` + `daemon.RestartWithBinary` (§III Wrap, Don't Reinvent); it CANNOT run in-process because the restart kills the serving process mid-request, so `POST /api/update` responds `202` then `exec.Command(self, "update")` with `SysProcAttr{Setsid: true}` (own session/process group, log → `~/.rk/update.log`) — **deliberately NOT `exec.CommandContext`-bound** because a detached child must outlive the server and §I's timeout rule only guards against a subprocess *blocking* the server, which a detached spawn cannot. *Why extract `internal/selfpath`*: the brew marker + self-path resolver were duplicated byte-for-byte in `upgrade.go` and `update.go` (unreachable cross-package `package main`/`package api`) — the shared package means brew-install detection cannot drift between the CLI and web entry points (cycle-1 review should-fix). *Accepted*: the release-vs-tap-formula CI race (a click in the few-minute window before the formula lands yields "already up to date" — no formula pre-verification); no in-flight lock (a second click harmlessly spawns another `rk update`); the write-only `Server.version` field (a review should-fix on new code, not a deletion candidate). *Rejected*: in-process upgrade (impossible — restart kills the server); `GET /api/version` (needless surface); a semver dependency; Web Push for update notices (reserved for waiting agents — update notices must not buzz phones, user-decided). Frontend: `useUpdateNotification()` hook + top-bar L3 UpdateChip + palette parity — see [ui-patterns](/run-kit/ui-patterns.md) § Update Notification. (`260713-4zap-update-notify-one-click-upgrade`)
- **`currentWindowEverSeen` gate on kill-redirect over unconditional "gone → redirect"** — The redirect in `app.tsx` that fires when `currentSession` or `currentWindow` is null treats every SSE payload as authoritative by default, but the first payload after a navigation is not: the SSE hub can replay a cached snapshot before delivering fresh data, and tmux window-list enumeration can lag by one poll tick when a session was just created. Unconditionally redirecting on either transient state tears down `TerminalClient` mid-mount. The gate (a ref keyed on `${server}|${session}|${window}`, flipped true the first time `currentWindow` resolves non-null, reset on URL change) defers the redirect until the URL target has been observed as valid at least once. Rejected: inspecting SSE payload identity or delaying `isConnected` (both push complexity into the hub layer and would not distinguish real kills from stale catch-up). Rejected: adding a fixed startup debounce (arbitrary delay, races with fast tests). The gate preserves fast-redirect behavior for genuine kills (seen flag stays true after first valid observation) while eliminating the startup race. (`260417-g9uo-mobile-tap-focus-regression`)
- **CSS Grid topology for the rotated shell over nested flex** — The shell rotation makes the sidebar a column that spans all three rows (topbar, content, bottombar). Expressing "sidebar full-height; right column is a 3-row stack" with nested flex requires either a sibling sidebar with explicit height management plus a flex-col right column, or a 2-row layout with sidebar nested inside row 1 — neither matches the visual model and both diverge as future cells are added. Grid encodes the topology directly via `grid-template-areas: "sidebar topbar" "sidebar content" "sidebar bottombar"`; collapse becomes a one-line column-width change (`${sidebarWidth}px 1fr` ↔ `0 1fr`) animatable as a single `grid-template-columns` transition. Modern browsers (Safari 14+, Chrome 100+) animate `grid-template-columns` reliably. (`260509-17m3-rotated-shell-layout`)
- **Sidebar collapses to 0px, not a 48px rail** — Maximum content width when collapsed, accepting a small visual layout shift on toggle (the hamburger appears to "move" from sidebar-adjacent to page-adjacent, though structurally it always lives at TopBar.left — see next decision). Rejected: a 48px icon rail (VS Code/Linear/Slack convention) — avoids the visual relocation but consumes content width for a usability gain that diverges from the user's stated preference for max content width. (`260509-17m3-rotated-shell-layout`)
- **Hamburger statically rendered at TopBar.left** — Makes the visual relocation on collapse a free side effect of the grid column collapsing — no component-level relocation logic, no conditional mount/unmount, no animation orchestration. The hamburger's home is always TopBar.left; the column is what changes. Rejected: state-conditional placement (hamburger inside sidebar when open, inside topbar when closed) — adds a state-dependent component-tree mutation that complicates focus management and adds animation responsibility. (`260509-17m3-rotated-shell-layout`)
- **Single `sidebarOpen` boolean over `sidebarOpen + drawerOpen`** — The pre-rotation distinction between "sidebar open on desktop" and "drawer open on mobile" was a rendering-form difference, not a state difference. The user's intent ("show me the sidebar") is one boolean; the renderer picks the form (grid column vs absolute overlay) by viewport width. Collapsing the two booleans simplifies persistence (one localStorage key, `runkit-sidebar-open`), removes a class of "they're out of sync" bugs, and makes the keyboard chord a single state mutation. Rejected: keep two booleans — preserves a historical artifact at the cost of doubled persistence/reset logic and a confusing surface for new contributors. (`260509-17m3-rotated-shell-layout`)
- **`<Shell>` as a shared wrapper over per-route shell components** — Both AppShell and BoardPage need the identical 3-row × 2-column grid, sidebar collapse, mobile overlay, and visual-viewport handling. A single `<Shell>` (`src/components/shell/shell.tsx`) lets both routes consume a uniform topology by wrapping children in `style={{ gridArea: "sidebar" | "topbar" | "content" | "bottombar" }}` elements; collapse animation, mobile overlay, and the topbar/content/bottombar geometry stay in one place. Rejected: per-route shell components — guaranteed to diverge over time (different collapse timings, drift in mobile overlay handling), defeats the rotation's "one topology, many routes" goal. Pattern note: future routes that need the same chrome should adopt `<Shell>` rather than reimplementing the grid. (`260509-17m3-rotated-shell-layout`)
- **`FocusedTerminalContext` as a separate, dedicated context (not a slice of ChromeContext)** — When BottomBar moved up to shell-level, it needed a way to target the focused terminal regardless of route. Folding the focused-terminal target into ChromeContext would mix UI state (sidebar open, fixed-width) with terminal-WS state and force every Chrome consumer to re-render whenever focus changes. A small dedicated context with its own provider (`FocusedTerminalProvider` mounted in `RootWrapper`) keeps the dependency narrow: only BottomBar and the ComposeBuffer-rendering TerminalClient subscribe. Compose state (`composeOpen` / `setComposeOpen`) was lifted to the same context so the shell-level BottomBar can open compose for the focused pane on the board route — alternative was lifting `<ComposeBuffer>` itself to shell level, which would have required rewiring the file-upload chain (`composeFiles`/`composeInitialText`/`useFileUpload`) across 3-4 components. Frozen-target semantics are preserved because `compose-buffer.tsx` already snapshots `wsRef` on mount; only the focused pane's `TerminalClient` renders ComposeBuffer at any time. Pattern note: future multi-input UIs that need a single shared input target should follow this "lift focused-input target to context" pattern rather than prop-drilling. (`260509-17m3-rotated-shell-layout`)
- **Read-only setter pair for in-memory + persisted state (`setSidebarWidth` + `persistSidebarWidth`)** — Drag-resize fires per-pointermove (potentially 60+ events per second). Persisting on every event would write localStorage 60 times per second per drag — unnecessary I/O and cache thrash. Splitting the dispatch into `setSidebarWidth(width)` (in-memory state) and `persistSidebarWidth(width)` (localStorage write) lets the drag handler call the cheap one in the loop and the expensive one once at drag-end. Convention note: future state with the same shape (live in-memory updates + final commit to persistence) should follow this two-setter pattern rather than debouncing inside the dispatch (which couples state lifecycle to a timing heuristic). (`260509-17m3-rotated-shell-layout`)
- **Mobile sidebar overlay positioned via grid (`gridRow: "2/4"`), not `fixed inset-0`** — The spec's literal phrasing was `fixed inset-y-0 left-0 z-50`, but the project convention recorded in `fab/project/context.md` and the existing `mobile-layout.spec.ts` assertion (`sidebarBox.y > 0`) requires the mobile overlay to leave the topbar visible — the topbar has the hamburger which closes the overlay, so covering it would be a usability regression. The implementation uses a grid-row-spanning child with `position: absolute` (not `fixed`) so the overlay anchors below the topbar inside the Shell. Pragmatic resolution: spec-wording "fixed-position overlay with backdrop" is satisfied — the overlay is fixed-position-style (absolute inside the Shell that itself is full-viewport), it has a backdrop, it carries `role="dialog" aria-modal="true"`. (`260509-17m3-rotated-shell-layout`)
- **Known follow-up — `sidebarChildren` prop duplication on `<Shell>`** — Both `AppShell` and `BoardPage` build the sidebar JSX once into a local `sidebarElement`, then pass it as `<Shell sidebarChildren={...}>` (used on mobile overlay) AND wrap it in a desktop-branch `<aside style={{gridArea:"sidebar"}}>{sidebarElement}</aside>` (used on desktop). This duplicates a small bit of JSX at every Shell consumer. Considered alternatives: a render-prop or a single named "sidebar" slot owned by Shell. The desktop branch in AppShell also includes the drag-handle aside wrapper, which BoardPage does NOT need — collapsing the duplication requires either (a) Shell knowing about the drag handle (small abstraction leak) or (b) exposing a separate "desktopChrome" slot. Captured here as a deferred refactor — not in 17m3's scope. (`260509-17m3-rotated-shell-layout`)

## Security

- All subprocess calls use `os/exec.CommandContext` with argument slices (never `sh -c` or shell strings)
- All `exec.CommandContext` calls include timeout via `context.WithTimeout` (10s tmux, 30s build). Terminal relay attach uses `context.WithCancel` (long-lived, cancelled on disconnect)
- User input validated via `internal/validate` before reaching any subprocess
- Directory listing restricted to `$HOME` via `ExpandTilde()` — rejects `..` traversal, absolute paths outside home, and `~username` syntax. Symlinks under `$HOME` are not resolved (accepted risk for local dev tool)
- File uploads: filename sanitized via `SanitizeFilename()` (strips path separators, null bytes, leading dots, collapses dot sequences); 50MB size limit enforced server-side via `http.MaxBytesReader`; writes via `os.Create` (not subprocess)
- CORS: permissive by default (`*` origin) for multi-client API flexibility. TLS handled by Tailscale Serve in production
- Web Push: the VAPID private key (`~/.rk/vapid.json`) is written mode `0600` and is **never** returned to any client — only the public key is served via `GET /api/push/vapid-public-key`. The subscribe/notify endpoints validate JSON shape and bound the request body (64 KiB); no subprocess execution is introduced by the push path (`260615-xd9r-web-push-notifications`)
