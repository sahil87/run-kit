### Homebrew Formula: `run-kit` command alias

#### R1: Formula installs `run-kit` as a symlink alias of `rk`
The `def install` block in `.github/formula-template.rb` MUST install a `run-kit` command symlink pointing at the installed `rk` binary, in addition to the existing `rk` install, using the standard Homebrew `bin.install_symlink` idiom. The `rk` binary MUST remain the canonical installed artifact (no rename), and no second binary is shipped.

- **GIVEN** the formula is generated by CI from `.github/formula-template.rb` and installed via `brew install sahil87/tap/rk`
- **WHEN** the `install` method runs
- **THEN** `bin/rk` is installed as today AND `bin/run-kit` exists as a symlink resolving to `bin/rk`
- **AND** invoking `run-kit <subcommand>` runs the same binary as `rk <subcommand>` (the binary is invocation-name-agnostic — resolves itself via `os.Executable()` + `EvalSymlinks`, per intake Assumption 6)

#### R2: Formula test asserts the `run-kit` alias resolves and runs
The formula's `test do` block in `.github/formula-template.rb` MUST assert that the `run-kit` alias runs and prints the version via the `--version` flag — the `version` subcommand was removed in PR #90 (`rk version` exits 1 with `unknown command`; `--version`/`-v` is the only version surface, verified empirically against the installed binary). The pre-existing `rk version` assertion carries the same latent stale-subcommand defect and MUST be corrected to `--version` in the same block — Homebrew's `shell_output` asserts exit status 0, so any subcommand-form assertion fails `brew test`. This is the only executable test seam this change has (it runs at `brew install`/`brew test` time), satisfying the code-quality "must include tests" principle.

- **GIVEN** the installed formula (with the R1 symlink present)
- **WHEN** `brew test rk` runs the `test do` block
- **THEN** `run-kit --version` exits 0 and its output matches `"rk version"` (the version banner is keyed on the static Cobra `Use: "rk"`, so both commands print the same `rk version v…` string)
- **AND** the corrected `rk --version` assertion passes alongside it

#### R3: Docs mention the `run-kit` alias at the install touchpoint
`docs/site/install.md` § Install and `README.md` (at the install/quickstart snippet) MUST each mention, once, that the Homebrew formula also installs `run-kit` as an interchangeable alias of `rk`. The mention MUST be additive (one sentence) — it is NOT a rebrand; existing `rk`-keyed prose, the `brew install sahil87/tap/rk` command, and command tables stay as-is.

- **GIVEN** a reader following the install instructions
- **WHEN** they read the install step in `docs/site/install.md` or the README quick-start
- **THEN** they learn that `run-kit` works as an interchangeable alias of `rk`
- **AND** no existing install/upgrade instruction, command name, or the formula-install command line is renamed or removed

### Non-Goals

- **No binary rename** — release artifact names (`rk-<os>-<arch>.tar.gz`), tarball contents (single `rk` member), the formula name (`brew install sahil87/tap/rk`), and `upgrade.go`'s `/Cellar/rk/` detection marker all stay keyed on `rk`.
- **No Go code changes** — the binary is already invocation-name-agnostic (`os.Executable()` + `EvalSymlinks` everywhere; verified in intake Assumption 6). Cobra `Use: "rk"` stays static — `run-kit --help` prints usage as `rk [command]` (intake Assumption 4).
- **No CI workflow changes** — the cross-compile step, tarball layout, and `sed` substitution in `.github/workflows/release.yml` are untouched; the template flows through the existing "Update Homebrew tap" step unchanged.
- **No tap formula alias** (`brew install sahil87/tap/run-kit`) — that is an `Aliases/` entry in the separate `sahil87/homebrew-tap` repo; possible follow-up, out of scope.
- **No dev-build symlink** — `just build` keeps producing only `bin/rk`.
- **No shell-alias approach** — wouldn't work in scripts or hooks.

### Design Decisions

1. **`rk` stays canonical, `run-kit` is the symlink alias**: invert the user's literal "make rk an alias of run-kit" phrasing — *Why*: outcome is identical (both commands work) and keeping `rk` canonical avoids churning release artifact names, tarball contents, the formula name, and the `/Cellar/rk/` upgrade marker for zero user-visible gain — *Rejected*: renaming the binary to `run-kit` (cascades through release artifacts + upgrade detection). (Intake Assumption 1.)
2. **Homebrew `bin.install_symlink` mechanism**: a formula symlink, not a second shipped binary and not a shell alias — *Why*: standard Homebrew idiom, works in scripts/hooks/non-interactive shells, needs no artifact or binary changes — *Rejected*: shell alias (doesn't work in scripts), second binary (doubles artifact size for nothing). (Intake Assumption 2.)
