# run-kit Architecture

## System Overview

run-kit is a web-based agent orchestration dashboard. In production, a single Go binary runs as a daemon in a dedicated tmux session:

1. **Go backend** (`app/backend/`, default port 3000) — single binary serving REST API, SSE, WebSocket terminal relay, and SPA static files on one port. Cobra CLI with subcommands: `serve` (default, with `-d`/`--restart`/`--stop` daemon flags), `update` (alias: `upgrade`), `doctor`, `status`, `context`, `init-conf`, `riff`. Version info via `--version`/`-v` global flag (Cobra built-in)

In development, `just dev` runs two concurrent processes:
- Vite dev server (`:RK_PORT`, default 3000) — HMR, proxies `/api/*`, `/relay/*`, and `/proxy/*` to Go backend
- Go backend (`:RK_PORT+1`, default 3001) — API, WebSocket relay, SPA static serving

Configuration via env vars: `.env` (committed) defines `RK_PORT` and `RK_HOST`, `.env.local` (gitignored) for overrides. Go backend and Vite both read `RK_PORT`/`RK_HOST` directly — no intermediate `BACKEND_*` vars. In dev mode, `dev.sh` is the translation layer: it passes `RK_PORT+1` to the Go backend and `RK_PORT` to Vite. `dev.sh` accepts `--port` for ad-hoc overrides. Tmux config defaults to `~/.rk/tmux.conf` (scaffolded by `rk init-conf`), overridable via `RK_TMUX_CONF` env var. The config is embedded in the binary via `go:embed` and written to disk by `init-conf`. All consumers use `tmux.ConfigPath()` getter — no direct env var reads elsewhere. User extensions go in `~/.rk/tmux.d/*.conf` — a drop-in directory sourced via `source-file -q` at the end of the default config. Files load in lexicographic order (numeric prefixes control precedence). The `-q` flag silences errors when the directory is empty or missing.

run-kit connects to **one tmux server at a time**, determined by the URL path segment. The URL hierarchy is `/$server/$session/$window` — the server name is always in the URL, making URLs fully shareable and self-contained. The active server is sent as a `?server=` query parameter on every API request (SSE, REST, WebSocket) by the frontend's `withServer()` helper. The backend is stateless — it defaults to the `default` tmux server when the parameter is absent. Server discovery scans `/tmp/tmux-{uid}/` for socket files. The frontend persists the last-used server in localStorage (`runkit-server`) as a convenience for the server list page, but the URL is always the source of truth. Server lifecycle: create (via server list page or command palette), kill (`tmux kill-server`), switch (navigate to `/$newserver`).

The pane-boards view (`/board/$name`) is a deliberate exception: it is a root-level peer to `/$server` (NOT under `/$server/...`) because boards aggregate windows across multiple tmux servers. It does not render the server-scoped AppShell — see `## Boards Feature` below.

## Data Model

**No database.** State derived at request time from:
- **tmux server** — `tmux list-sessions`, `tmux list-windows` via `internal/tmux/tmux.go`. Project roots derived from window 0's `pane_current_path`. Read-path functions (`ListSessions`, `ListWindows`, `ListServers`) accept `context.Context` from HTTP handlers for request cancellation propagation. Mutation functions (Create/Kill/Rename) use `context.Background()` — user-initiated actions complete regardless of client disconnect
- **`run-kit.yaml`** — per-project config file at the git root of a session's working directory. Currently stores `session_color: N` (ANSI palette index 0-15). Read best-effort by `config.ReadSessionColor()` — missing file or missing key means no color, never an error. Written by `config.WriteSessionColor()` — clearing removes the key (deletes the file if it becomes empty, preserves other keys if present). Git root resolved via `config.FindGitRoot()` (walks up from `pane_current_path` looking for `.git`). Gitignored by convention. Not server config (env vars handle that per constitution) — this is per-project user preferences
- **tmux `@color` user option** — per-window color stored as a tmux user option (`#{@color}` in `ListWindows()` format string). Ephemeral — survives session lifetime but not tmux server restarts. Set via `tmux set-option -w -t "{session}:{index}" @color {N}`, cleared via `tmux set-option -wu`. Parsed into `WindowInfo.Color *int` (nil when unset)
- **tmux `@rk_session_order` server-scoped user option** — sidebar session render order stored as a JSON-encoded array of session names on the tmux *server* (not per-window). Survives the same lifetime as the tmux server (lost on tmux server death, NOT on rk-go restart per Constitution VI). Read by `tmux.GetSessionOrder(ctx, server)` — treats unset / no-server / `failed to connect` as empty slice; surfaces other errors (including JSON decode) as wrapped Go errors. Written by `tmux.SetSessionOrder(ctx, server, order)` — `nil` slice normalized to `[]` so round-trips are lossless. Both functions go through `tmuxExecRawServer` with `context.WithTimeout(ctx, TmuxTimeout)` (10s). The constant `tmux.SessionOrderOption = "@rk_session_order"` is the canonical option name
- **fab pane map** — `internal/sessions` invokes the `fab` router on PATH via `exec.CommandContext(ctx, "fab", "pane", "map", "--json", "--all-sessions")` with `cmd.Dir = repoRoot` so the router can resolve the project's `fab_version` from `fab/project/config.yaml`. Returns per-window fab state (change name, pipeline stage, agent state, idle duration) as a map keyed by `session:windowIndex` for O(1) lookup during result assembly. Split-pane collisions on that key are resolved by the extracted `dedupEntries(entries) map[string]paneMapEntry` helper, which applies priority Change > AgentState > first-seen — i.e. a change-bound pane always wins, otherwise a pane with populated `AgentState` wins, otherwise the first-seen entry is preserved. After the fab-kit v1.5.0 `_agents` schema refactor, `agentState`/`agentIdleDuration` populate independently of `fabChange`/`fabStage`, so discussion-mode agent panes (populated `AgentState`, nil `Change`) now win over bare sibling panes in split windows. Results cached at package level with 5s TTL (`fetchPaneMapCached`) — double-check pattern prevents thundering herd, stale cache preserved on fetch error for graceful degradation

### Performance Caching

Two in-memory caches reduce subprocess spawning on the SSE hot path (justified per code-quality.md performance carve-out — not general-purpose caching):

| Cache | Location | TTL | Key | Purpose |
|-------|----------|-----|-----|---------|
| SSE session cache | `sseHub.cache` (`api/sse.go`) | 500ms | server name | Skip `FetchSessions()` subprocess calls when data is fresh |
| Pane-map cache | package-level (`internal/sessions`) | 5s | (single entry) | Skip `fab pane map` subprocess calls between SSE ticks |

Both caches use `sync.RWMutex` for thread safety. The SSE hub's existing JSON diff-check (`previousJSON`) remains as a separate concern — the session cache prevents redundant subprocess spawning, while the diff prevents redundant SSE broadcasts

## Backend Libraries (Go Modules)

Packages in `app/backend/internal/`:

| Package | Responsibility |
|---------|---------------|
| `internal/tmux` | All tmux operations via `os/exec.CommandContext` with argument slices + `context.WithTimeout` (10s). Commands target the dedicated `runkit` server via `-L runkit` prefix (built by `runkitPrefix()`); config path defaults to `~/.rk/tmux.conf`, overridable via `RK_TMUX_CONF` env var, resolved to absolute path at init. `ConfigPath()` getter exposes the resolved path. `OriginalTMUX` package-level var captures `$TMUX` before `init()` strips it (via `os.Unsetenv("TMUX")`) — used by `cmd/rk/context.go` to restore `$TMUX` in child process environments when querying the pane's own tmux server. `DefaultConfigBytes()` returns the embedded `internal/tmux/tmux.conf` content (via `go:embed`). Read-path functions accept `context.Context` for request cancellation: `ListSessions(ctx, server)`, `ListWindows(ctx, session, server)`, `ListServers(ctx)`. `ListServers` probes sockets in parallel using a bounded goroutine pool (cap 10, semaphore channel) — N dead sockets cost ~2s instead of 2*N seconds. Mutation functions (`CreateSession`, `KillSession`, `RenameSession`, `CreateWindow`, `KillWindow`, `SwapWindow`, `RenameWindow`, `SendKeys`, `SelectWindow`, `SplitWindow`, `KillActivePane`, `KillPane`, `SetWindowColor`, `UnsetWindowColor`) use `withTimeout()` / `context.Background()`. `ListSessions(ctx, server)` queries the specified tmux server, returning `SessionInfo` structs with a `Name` field. `ReloadConfig(server)` hot-reloads the tmux config via `source-file` on the specified server. `ListWindows()` includes `isActiveWindow` flag from `#{window_active}`, `PaneCommand` from `#{pane_current_command}`, and raw `ActivityTimestamp` from `#{window_activity}`. After calling `list-windows`, `ListWindows()` issues a second `list-panes -s -t <session>` call to populate `Panes []PaneInfo` on each `WindowInfo`; pane failure is non-fatal (windows returned with empty `Panes`). `WorktreePath` continues to be sourced from `#{pane_current_path}` in the `list-windows` format string — `list-panes` is additive. `PaneInfo` struct: `PaneID string json:"paneId"`, `PaneIndex int json:"paneIndex"`, `Cwd string json:"cwd"`, `Command string json:"command"`, `IsActive bool json:"isActive"`. Package-level `paneFormat` var joins `#{window_index}`, `#{pane_id}`, `#{pane_index}`, `#{pane_current_path}`, `#{pane_current_command}`, `#{pane_active}` with `listDelim`. `parsePanes(lines []string) map[int][]PaneInfo` parses 6-field tab-delimited `list-panes` output, grouping by window index (field 0); malformed lines silently skipped; returns nil on empty/all-malformed input. `WindowInfo` struct uses `FabChange`/`FabStage` fields, plus `AgentState`/`AgentIdleDuration` (populated by pane-map enrichment in sessions package), plus `Color *int json:"color,omitempty"` (ANSI palette index from tmux `@color` user option, nil when unset), plus `Panes []PaneInfo json:"panes,omitempty"`, plus `RkType string json:"rkType,omitempty"` and `RkUrl string json:"rkUrl,omitempty"` for iframe window metadata (populated from tmux user-defined options `@rk_type` and `@rk_url` via `ListWindows` format string). `EnsureConfig()` writes the embedded default config to `~/.rk/tmux.conf` if absent and creates `~/.rk/tmux.d/` (called at serve startup; directory creation runs even when config already exists). `ListKeys(server)` runs `tmux list-keys` on a server and returns raw output lines (returns nil on "no server running"). The `-f configPath` flag is scoped to `CreateSession` and `ReloadConfig` only via `configArgs()` — not passed on every command. Both `tmuxExec` and `tmuxExecDefault` capture stderr in error messages for diagnostics. `SetWindowOption(ctx, session, index, server, option, value)` sets a user-defined window option via `tmux set-option -w`. `CreateWindowWithOptions(session, name, cwd, server, options)` creates a window and atomically sets user-defined options via a single `\;`-chained tmux command (prevents SSE from seeing the window before metadata is set). `ListWindows` format string includes `#{@color}` as field 8, `#{@rk_type}` and `#{@rk_url}` as fields 9-10; `parseWindows` extracts them (empty string when unset) |
| `internal/sessions` | `FetchSessions(ctx, server)` lists sessions and fetches windows for all of them in parallel for the requested tmux server `server` (passing `server` to `ListWindows`), then enriches with fab state. `ProjectSession` struct has `Name`, `SessionColor *int json:"sessionColor,omitempty"`, and `Windows` fields. `SessionColor` is read from `run-kit.yaml` at the git root (via `config.FindGitRoot` + `config.ReadSessionColor`) during session assembly — nil when no config or no `session_color` key. Per-window enrichment model: pane-map returns per-pane fab state, joined to windows by `session:windowIndex` key. `paneMapEntry` struct uses `*string` for nullable JSON fields (change, stage, agent_state, agent_idle_duration). `fetchPaneMapCached(server, repoRoot)` wraps `fetchPaneMap(server, repoRoot)` with 5s TTL cache (package-level `sync.RWMutex`, double-check pattern). On cache miss, calls `fetchPaneMap(server, repoRoot)` which invokes the `fab` router on PATH (`fab pane map --json --all-sessions`) with `cmd.Dir = repoRoot` and a 10s context timeout; the router resolves the project's `fab_version` from `fab/project/config.yaml`. The extracted `dedupEntries(entries) map[string]paneMapEntry` helper collapses split-pane collisions with priority Change > AgentState > first-seen, so discussion-mode windows (agent running with no active change) can populate `AgentState` independently of `Change` and still win over bare sibling panes after the fab-kit v1.5.0 `_agents` schema refactor. `findRepoRoot(dir)` walks up from `dir` looking for `fab/project/config.yaml` (the fab-project identity marker, per `fab preflight`) and returns that directory, or `""` if not found. Stale cache entry preserved on fetch error for graceful degradation; if pane-map fails on cold start, all windows get empty fab fields |
| `internal/validate` | Input validation for names/paths + tilde expansion with `$HOME` security boundary + filename sanitization for uploads |
| `internal/config` | Server config (port, host) — reads `RK_PORT` and `RK_HOST` env vars with defaults (3000, 127.0.0.1). Also provides `run-kit.yaml` operations via `internal/config/runkit_yaml.go`: `FindGitRoot(dir string) string` walks up from dir to find `.git` (returns `""` if not found), `ReadSessionColor(projectRoot string) *int` reads `session_color` from `run-kit.yaml` (best-effort, nil on any error), `WriteSessionColor(projectRoot string, color *int) error` writes/clears the key (deletes file if empty, preserves other keys). Uses simple line-by-line YAML parsing via `splitYAMLLine` — no yaml.v3 dependency |
| `internal/daemon` | Daemon lifecycle management — `IsRunning()`, `Start()`, `Stop()`, `Restart()`, plus `StartWithBinary(binPath)` / `RestartWithBinary(binPath)` (used by `cmd/rk/upgrade.go` after `brew upgrade` when the running process's `os.Executable()` path is stale). Manages `rk serve` in a dedicated tmux server (socket `rk-daemon`, session `rk-daemon`, legacy session `rk` consulted by `Stop`/`Restart` for transparent upgrade, window `serve`). All tmux calls go through `runTmux` which uses `exec.CommandContext` with 5s timeout and prepends `-L rk-daemon`. Added port-based liveness probe via `net.DialTimeout` (400ms, with loopback substitution for wildcard hosts) and stale-socket reaping via `runTmux(ctx, "kill-server")` before session creation; supports daemon startup logging tee via `RK_DAEMON_LOG` env var (exported as `daemon.LogEnvVar`). Used by `serve.go` (CLI flags) and `upgrade.go` (auto-restart after update) |
| `internal/metrics` | Host-level system metrics collector reading Linux procfs. `Collector` struct with `sync.RWMutex`-protected `MetricsSnapshot`. `NewCollector(pollInterval)` creates collector, caches hostname via `os.Hostname()`, initializes 60-entry CPU ring buffer with zeros. `Start(ctx)` launches background poll goroutine (exits on `ctx.Done()`). `Snapshot()` returns a deep-copied `MetricsSnapshot` under `RLock`. Collects: CPU usage from `/proc/stat` (delta-based, ring buffer), memory from `/proc/meminfo` (`MemTotal` - `MemAvailable`), load from `/proc/loadavg`, disk via `syscall.Statfs("/")`, uptime from `/proc/uptime`. CPU core count from `/proc/stat` `cpu\d+` line count. All procfs readers return zero values on non-Linux (graceful degradation, no crash). `MetricsSnapshot` struct contains `Hostname`, `CPU` (samples `[]float64`, current, cores), `Memory` (used/total bytes), `Load` (avg1/5/15, cpus), `Disk` (used/total bytes), `UptimeSecs` |
| `internal/settings` | Global user settings persisted at `~/.rk/settings.yaml`. `Settings` struct with `Theme string` field. `Default()` returns `Settings{Theme: "system"}`. `Load()` reads the settings file, returns `Default()` if missing. `Save(s Settings)` writes the file, creating `~/.rk/` (mode 0755) if absent. Uses simple `key: value` text parsing (not yaml.v3) — one field, no heavy dependency needed |

## API Layer

All endpoints served by the single Go binary on one port. POST-only mutations with path-based intent (no multiplexed action field).

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/health` | GET | Returns `200 {"status":"ok","hostname":"..."}` for health checks. Hostname computed once at startup via `os.Hostname()`, stored in `Server` struct; falls back to `""` on error |
| `/api/sessions` | GET | Returns `ProjectSession[]` — one per tmux session, with auto-detected fab enrichment (`fabChange`/`fabStage` on windows) |
| `/api/sessions` | POST | Create session — JSON body `{"name":"...","cwd":"..."}`. Returns `201 {"ok":true}` |
| `/api/sessions/order` | GET | Returns `{"order":[...]}` — JSON array of session names representing the user-defined sidebar render order on `?server=`. Reads tmux server-scoped user-option `@rk_session_order`. Unset/no-server cases return `200 {"order":[]}` (not 404); other tmux failures (e.g., malformed JSON in stored option) return `500` with error body |
| `/api/sessions/order` | PUT | Persists the sidebar session order — JSON body `{"order":[...]}`. Each name validated via `validate.ValidateName`. JSON-encoded value written to tmux user-option `@rk_session_order` via `set-option -s` (server-scoped). After a successful write, broadcasts `event: session-order` (payload `{server, order}`) synchronously to all SSE clients on that server. Stale names (not currently a live session) are accepted — frontend's `orderedSessions` sort tolerates them. Returns `400` on malformed JSON or invalid name; `200 {"ok":true}` on success |
| `/api/sessions/:session/kill` | POST | Kill session — `:session` validated via `validate.ValidateName()`. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows` | POST | Create window — JSON body `{"name":"...","cwd":"..."}`. Returns `201 {"ok":true}` |
| `/api/sessions/:session/windows/:index/kill` | POST | Kill window — `:index` must be non-negative integer. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/move` | POST | Move (swap) window — JSON body `{"targetIndex": N}` where N is a non-negative integer. Calls `tmux swap-window` to exchange source and target window positions. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/move-to-session` | POST | Move window to another session — JSON body `{"targetSession":"..."}`. Calls `tmux move-window` to move the source window into the target session (tmux auto-assigns index). Returns `200 {"ok":true}` |
| `/api/sessions/:session/color` | POST | Set or clear session color — JSON body `{"color": N}` (0-15) to set, `{"color": null}` to clear. Writes `session_color` to `run-kit.yaml` at the session's git root (derived from window 0's `WorktreePath` via `config.FindGitRoot`). Validates session name and color range. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/color` | POST | Set or clear window color — JSON body `{"color": N}` (0-15) to set, `{"color": null}` to clear. Executes `tmux set-option -w @color` (set) or `tmux set-option -wu @color` (clear). Validates session name, window index, and color range. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/rename` | POST | Rename window — JSON body `{"name":"..."}`. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/url` | PUT | Update iframe window URL — JSON body `{"url":"..."}` (non-empty after trim). Runs `tmux set-option -w @rk_url`. Returns `200 {"ok":true}` |
| `/api/sessions/:session/windows/:index/keys` | POST | Send keys — JSON body `{"keys":"..."}` (non-empty after trim). Returns `200 {"ok":true}` |
| `/api/directories` | GET | Server-side directory listing for autocomplete — `?prefix=~/code/wvr` returns matching dirs under `$HOME` |
| `/api/sessions/:session/upload` | POST | File upload — session from URL path (not form field). Multipart with `file` field, optional `window` field (defaults to `"0"`). Resolves project root via `ListWindows`, writes to `.uploads/{timestamp}-{name}`, auto-manages `.gitignore`. 50MB limit. Returns `200 {"ok":true,"path":"..."}` |
| `/api/sessions/stream` | GET | SSE — hub singleton polls tmux every 2.5s, fans out full snapshots to all connected clients on change. Three event types: `event: sessions` (per-server, deduplicated), `event: metrics` (server-independent, every tick), and `event: session-order` (per-server, eager — broadcast synchronously on PUT to `/api/sessions/order`, plus once per server on first poll as a bootstrap so the order survives an rk-go restart that left tmux running). Session-order payload shape: `{"server":"<name>","order":[...]}`. Deduplicates polling across browser tabs. 30-minute lifetime cap per connection. Hub clients map is `map[string][]*sseClient` keyed by server name for O(1) server-scoped broadcast. `sync.RWMutex` — `RLock()` for read-only operations in `poll()` (collecting server keys, counting clients), `Lock()` for writes (broadcasting, updating `previousJSON`/`previousOrderJSON`). Channel buffer 32. Drop logging via `slog.Warn` with per-client boolean debounce (resets on successful send). Hub holds a `*metrics.Collector` reference; on each tick calls `Snapshot()` and broadcasts `event: metrics` with JSON payload to all clients regardless of server. The hub also holds a `SessionOrderFetcher` (defaults to `prodSessionOrderFetcher{}` calling `tmux.GetSessionOrder`) used for the bootstrap; tests substitute via `hub.orderFetcher = stub`. New clients receive whichever cached snapshots are populated at connect time (`sessions`, `session-order`, `metrics`), in that order. On a warm hub these arrive immediately. On a cold hub (rk-go just started, no client has connected for that server yet, no PUT has fired), the cache for that server is empty and the first client gets only what's already populated; the missing entries arrive on the first poll tick — for `session-order` specifically, the once-per-server `poll()` bootstrap reads tmux and broadcasts. Cached payloads kept in `previousJSON` / `previousOrderJSON` / `cachedMetricsJSON` |
| `/api/tmux/reload-config` | POST | Reload tmux config — server from `?server=` param (default `"default"`). Runs `source-file` on the specified server. Returns `200 {"status":"ok"}` |
| `/api/servers` | GET | List available tmux servers — scans socket dir, returns JSON array of `{name, sessionCount}` objects sorted alphabetically. Session counts computed by concurrent fan-out over `tmux.ListSessions(ctx, server)` (WaitGroup + mutex-protected results map); per-server `ListSessions` errors are logged at warn and yield `sessionCount: 0` (no 5xx surfaces). Empty discovery returns `[]` (not `null`) |
| `/api/servers` | POST | Create tmux server — JSON body `{"name":"..."}`. Creates session "0" in `$HOME`. Returns `201 {"ok":true}` |
| `/api/servers/kill` | POST | Kill tmux server — JSON body `{"name":"..."}`. Returns `200 {"ok":true}` |
| `/api/keybindings` | GET | Curated tmux keybindings — runs `tmux list-keys` on `?server=`, filters through a whitelist map, returns JSON array `[{key, table, command, label}]`. Returns `[]` if server not running or on error |
| `/api/settings/theme` | GET | Returns `{"theme": "..."}` — reads via `settings.Load()`. Returns `"system"` when no settings file exists. Not per-server (no `?server=` param) |
| `/api/settings/theme` | PUT | Accepts `{"theme": "..."}` — writes via `settings.Save()`. Returns `{"status": "ok"}` on success, `400` if theme is empty. Not per-server |
| `/api/boards` | GET | Lists pane boards aggregated across every server returned by `tmux.ListServers(ctx)`. Returns `200 [{name, pinCount}]` (alphabetical by `name`); `[]` when empty (never `null`). Not per-server (boards span servers — no `?server=` query) |

## Terminal Relay

WebSocket endpoint at `/relay/{session}/{window}?server=runkit|default` on the same port as the API — no separate relay port. Uses `gorilla/websocket` for WebSocket handling and `creack/pty` for PTY allocation. Implementation in `app/backend/api/relay.go`. The `server` query param determines which tmux server to attach to (defaults to `runkit`).

Per connection:
1. Validates the real session exists via `ListWindows(session, server)` — returns WebSocket close code `4004` if session or window not found
2. Generates an ephemeral session name `rk-relay-<8 hex>` via `crypto/rand` (`newEphemeralRelayName` in `relay.go`); 4B namespace, never derived from user input (constitution I)
3. Creates a per-WebSocket grouped session via `s.tmux.NewGroupedSession(r.Context(), server, session, ephemeral)` (`tmux [-L server] new-session -d -s <ephemeral> -t <real>`) — sessions in the same group share window membership but maintain **independent active-window state**, so concurrent relays no longer steal each other's `select-window`
4. `defer s.tmux.KillSessionCtx(context.Background(), server, ephemeral)` reaps the ephemeral on every exit path. Cleanup uses `context.Background()` rather than `r.Context()` because the request context is already cancelled at cleanup time (the trigger for the defer)
5. Selects the target window on the **ephemeral** via `s.tmux.SelectWindow(ephemeral, winIdx, server)` — never on the real session
6. Spawns `tmux [-L server] attach-session -t <ephemeral>` via `creack/pty`. Attaching to the ephemeral (not the real session) is the linchpin of the fix
7. Relays I/O between WebSocket and pty (goroutine for pty→WS, main loop for WS→pty)
8. Handles resize messages (JSON `{"type":"resize","cols":N,"rows":N}`) via `pty.Setsize`
9. On disconnect: `sync.Once` cleanup cancels context, closes PTY, kills process; the deferred `KillSessionCtx` then reaps the ephemeral. PTY reader goroutine calls `cleanup()` (not `conn.Close()`) on read failure — eliminates concurrent WebSocket close race. WebSocket connection closed only by `defer conn.Close()` on the main goroutine

The grouped-session indirection fixes the central board bug — N board panes targeting different windows of the same real session now show distinct content — and as a side effect closes a latent multi-tab bug where two browser tabs viewing different windows of the same session yanked each other's active window.

**Startup sweep**: `rk serve` reaps orphan `rk-relay-*` sessions across every known tmux server before binding HTTP listeners (`sweepOrphanedRelaySessions` in `app/backend/cmd/rk/serve_sweep.go`, wired into `serveCmd.RunE` after `tmux.EnsureConfig()` and before the `ListenAndServe` goroutine, with a 30s bounded context). The sweep iterates `tmux.ListServers(ctx)` and calls `tmux.ListRawSessionNames(ctx, server)` (the unfiltered variant — the user-facing filter would hide the very ephemerals being reaped); per-server failures are logged and skipped without blocking startup. See `tmux-sessions.md` § Per-WebSocket Ephemeral Grouped Sessions for the full convention.

Client-side WebSocket reconnection: exponential backoff (1s, 2s, 4s, 8s, 16s, max 30s) on unexpected close. Shows `[reconnecting...]` in terminal. Re-sends resize on successful reconnect. Skips reconnect on component unmount. On close code `4004` (session/window not found): shows `[session not found]` and navigates to `/` instead of reconnecting. Terminal page connects via `ws://${location.host}/relay/{session}/{window}?server={runkit|default}` — same host, server param from session metadata.

## Daemon Lifecycle

The Go binary manages its own daemon lifecycle via `run-kit serve` flags. The daemon runs in a dedicated tmux server (socket `rk-daemon`, session `rk-daemon`, window `serve`) — separate from the `runkit` server used for agent sessions. The session name matches the socket name so the combined exact-match target (`=rk-daemon:=serve`) cannot collide with user-facing sessions via tmux prefix matching. `LegacySessionName = "rk"` is consulted only by `Stop`/`Restart` (via `runningSessionCtx`, which checks `SessionName` first then falls back to the legacy name) so a daemon started by a pre-rename binary can still be stopped cleanly during the first upgrade. Daemon helpers live in `internal/daemon/`.

- `run-kit serve -d` — start daemon (errors if already running)
- `run-kit serve --restart` — idempotent: stop existing daemon (C-c → wait 5s) then start new one; starts fresh if no daemon running
- `run-kit serve --stop` — graceful shutdown via C-c to the tmux pane
- `run-kit update` — auto-restarts daemon after successful `brew upgrade` using `RestartWithBinary(brewBinPath)` where `brewBinPath` is `<brewPrefix>/bin/rk` derived from the resolved Cellar path (resolves to the new Cellar version post-upgrade, avoiding stale `os.Executable()` path from the running process)

Binary resolution: `Start()` uses `os.Executable()` + `filepath.EvalSymlinks()` (works for `serve -d` / `--restart` where the running binary is valid). `StartWithBinary(path)` accepts an explicit binary path and resolves its symlinks (used by `update` after `brew upgrade` deletes the old Cellar directory). Both delegate to shared `startSession(exe)` for tmux session creation.

Detection: `tmux -L rk-daemon has-session -t =rk-daemon` (anchored `=` prefix forces exact-match — see PR #196). `runningSessionCtx` consults `=rk-daemon` first, then falls back to `=rk` for transparent stop/restart of pre-rename daemons. No polling loop, no signal files, no supervisor script.

**Port-based liveness probe**: Before `Start()`/`StartWithBinary()` create the tmux session — and AFTER the `IsRunning()` check passes — they call `guardPortAvailable()`, which loads `config.Load()` and runs `net.DialTimeout("tcp", host:port, 400ms)` against `RK_HOST:RK_PORT`. Wildcard/empty hosts (`""`, `"0.0.0.0"`, `"::"`, `"[::]"`, `"[::1]"`) substitute `127.0.0.1` via `probeHost()` — dialing `0.0.0.0` itself is platform-inconsistent and not a reliable liveness signal. If the dial succeeds (something is listening), `Start` returns an error containing the substrings `already serving on`, `not under the rk-daemon`, and `RK_PORT` so scripts can pattern-match. Cobra `RunE` propagates the non-zero exit. This closes the split-brain detection gap where a foreground `rk serve` was invisible to the tmux-only `IsRunning()` check.

**Stale-socket reaping**: When `IsRunning()` is false AND the port probe passes, `Start`/`StartWithBinary` call `reapStaleDaemonSocket(ctx)` which invokes `runTmux(ctx, "kill-server")` — scoped strictly to `serverSocket = "rk-daemon"` (never the agent-session `runkit` server). Idempotent: tmux's "no server running" surface on cold start is the common case. Errors are logged at `slog.Debug` and never block startup; if anything is genuinely broken, the subsequent `startSession` surfaces it. This self-heals the orphaned-socket state that arose when a previous inner serve died on a port collision before this fix shipped.

**Startup logging via `RK_DAEMON_LOG`**: `startSession` resolves `os.UserCacheDir()/rk/daemon.log` (e.g. `~/.cache/rk/daemon.log` on Linux, `~/Library/Caches/rk/daemon.log` on macOS) via `resolveDaemonLogPath()` and passes it to the inner `tmux new-session` via `-e RK_DAEMON_LOG=<path>`. The serve startup's `setupSlog(level)` reads the env var, `os.MkdirAll(filepath.Dir(p), 0o755)`, opens the file with `O_APPEND|O_CREATE|O_WRONLY` (mode 0o644), and wraps `io.MultiWriter(os.Stderr, file)` so slog output is teed to both the tmux pane (preserves `tmux attach` debugging) and a durable log file. Open failures fall back to stderr-only with one `slog.Warn("daemon log unavailable", ...)`; `UserCacheDir()` failure in `startSession` spawns the inner serve without the env var (single `slog.Warn` upstream). Logging is best-effort — never blocks daemon creation. Constitution §VI compliance: the log is a passive append-only artifact, no supervisor, no signal file, no liveness derivation from contents.

The exported constant `daemon.LogEnvVar = "RK_DAEMON_LOG"` is the single source of truth (consumed by both `internal/daemon/startSession` and `cmd/rk/serve.go/setupSlog`).

## Chrome Architecture

**Lazy-loaded components**: `CommandPalette`, `ThemeSelector`, and `CreateSessionDialog` are loaded via `React.lazy()` with dynamic `import()` (not static imports). Each uses `.then(m => ({ default: m.X }))` to re-wrap named exports for `React.lazy()` compatibility. Render sites wrapped in per-component `<Suspense fallback={null}>` boundaries. These components are overlays/dialogs that render conditionally — lazy loading keeps them out of the initial bundle for faster first paint.

**Shell wrapper (CSS Grid topology)** — `<Shell>` (`app/frontend/src/components/shell/shell.tsx`) is the shared chrome wrapper used by both `AppShell` and `BoardPage`. The shell is a single CSS Grid root carrying `height: var(--app-height, 100vh)` (preserves iOS keyboard handling via `useVisualViewport`). Desktop topology (`>= 640px`):

```
grid-template-areas:
  "sidebar topbar"
  "sidebar content"
  "sidebar bottombar"
grid-template-rows:    auto 1fr auto
grid-template-columns: ${sidebarWidth}px 1fr   (when sidebarOpen)
grid-template-columns: 0 1fr                   (when !sidebarOpen)
transition: grid-template-columns 150ms ease-out
```

Sidebar is full-height — it spans all three rows and owns the entire left column. Topbar/content/bottombar stack in the right column. Sidebar collapse zeroes the first grid column (no 48px rail) — the topbar/content/bottombar cells extend their left edge to page.left automatically; no JS-side conditional rendering or component mutation. The drag-resize handle is hidden when `sidebarOpen === false` (the only re-open affordance is the hamburger). Drag-handle wiring is AppShell-only — `BoardPage` does not need a drag-handle (the dimension lives in `ChromeContext` and is shared).

Mobile topology (`< 640px`): single column, `grid-template-areas: "topbar" / "content" / "bottombar"`. Sidebar renders as an overlay child of the Shell positioned via `gridRow: "2/4"` (spans content + bottombar, leaves the topbar visible). Backdrop and `<aside>` use `position: absolute` (NOT `fixed inset-0`) so the overlay starts below the topbar — matches the existing project convention recorded in `fab/project/context.md`. The aside carries `role="dialog" aria-modal="true"`. Sidebar overlay open/close is driven by the same `sidebarOpen` state — `drawerOpen` is removed from `ChromeContext`.

**Hamburger placement** — the hamburger icon is statically rendered at `TopBar.left` in both `sidebarOpen` states. Visual "relocation" on collapse is a side effect of the grid column collapsing; there is no component-level relocation logic. When `sidebarOpen === true`, the hamburger sits at the sidebar.right / topbar.left seam (pixel-identical to "attached to sidebar"). When `sidebarOpen === false`, it sits at page.left because the topbar grid cell has extended to page.left.

**Keyboard toggle** — `Cmd+\` (macOS) / `Ctrl+\` (Linux/Windows) toggles `sidebarOpen` from any route that mounts `<Shell>`. The chord is registered in the `KeyboardShortcuts` modal under "Toggle sidebar" per Constitution V (Keyboard-First). The chord is suppressed when focus is inside a real text input/textarea/contenteditable, but NOT when focus is on xterm.js's helper textarea (recognized by `target.closest('.xterm') != null`) — without the xterm exception, the most common focus state would silently swallow the toggle.

**Sidebar width persistence** — width state lives in `ChromeContext` (`sidebarWidth`). The dispatch surface is split: `setSidebarWidth(width)` updates in-memory state only (called per-pointermove during drag), and `persistSidebarWidth(width)` writes `localStorage["runkit-sidebar-width"]` (called once at drag-end). This explicit split avoids localStorage thrash on every pointer event. Callers like the AppShell drag handler track the last-seen width in a ref (`dragLastWidthRef`) and call `persistSidebarWidth` once in `handleEnd`.

**Routing branches inside the `content` grid area**:
- AppShell (`/$server/$session/$window`): `<TerminalClient>` (terminal window) | `<IframeWindow>` (iframe window) | `<Dashboard>` (no session/window params). The iframe-banner row above `TerminalClient` (when `currentWindow.rkUrl` exists) remains inside `content`.
- BoardPage (`/board/$name`): `DesktopRow` / `MobileCarousel` body (the existing horizontal-scroll panes), flush with `sidebar.right` (no left gutter).

No `max-w-4xl` constraint — all zones span full width.

**ThemeProvider** (`app/frontend/src/contexts/theme-context.tsx`) — outermost provider. Split into ThemeStateContext (preference + resolved theme) and ThemeActionsContext (setTheme). Three modes: system, light, dark. Preference persisted to `localStorage` key `runkit-theme`. System mode listens to `matchMedia("(prefers-color-scheme: dark)")` change events. Theme applied via `data-theme` attribute on `<html>`. Blocking inline script in `index.html` `<head>` reads localStorage and sets `data-theme` before first paint (no-flicker initialization).

**ChromeProvider** (`app/frontend/src/contexts/chrome-context.tsx`) — split into two React contexts: `ChromeStateContext` (read-only state: `currentSession`, `currentWindow`, `sidebarOpen`, `sidebarWidth`, `isConnected`, `fixedWidth`) and `ChromeDispatchContext` (stable setter functions: `setCurrentSession`, `setCurrentWindow`, `setSidebarOpen` — persists to `localStorage["runkit-sidebar-open"]`, `setSidebarWidth` — in-memory only, `persistSidebarWidth(width)` — writes localStorage on drag-end, `setIsConnected`, `toggleFixedWidth`). `drawerOpen`/`setDrawerOpen` were removed when the rotated shell collapsed the desktop-vs-mobile sidebar split into a single `sidebarOpen` boolean (the rendering form differs per breakpoint, the state does not). Chrome derives its content from the current selection — no slot injection. `setLine2Left`, `setLine2Right`, `setBottomBar` removed; top bar and bottom bar read the selection directly. `Breadcrumb` type includes optional `dropdownItems` for session/window switching.

**FocusedTerminalContext** (`app/frontend/src/contexts/focused-terminal-context.tsx`) — tracks the currently-focused terminal across the app so the shell-level `BottomBar` can target the right WebSocket regardless of route. Mounted in `RootWrapper` between `SessionProvider` and `OptimisticProvider`; provider tree order is `ThemeProvider > ChromeProvider > RootWrapper(SessionProvider > OptimisticProvider > FocusedTerminalProvider > Outlet)`. Shape:

```ts
type FocusedTerminal = {
  wsRef: React.RefObject<WebSocket | null>;
  server: string;
  session: string;
  windowIndex: string;
} | null;

type FocusedTerminalContextValue = {
  focused: FocusedTerminal;
  setFocused: (t: FocusedTerminal) => void;
  composeOpen: boolean;
  setComposeOpen: (open: boolean) => void;
};
```

Producers: `TerminalClient` registers on mount when its `registerFocus` prop is `true` (single-terminal route default) and clears on unmount. `BoardPane` registers when its `isFocused` prop becomes true (and on initial pane mount); it manages the registration itself and passes `registerFocus={false}` to its inner `TerminalClient` to avoid producer contention. `BoardPane` does NOT clear on focus loss — the next pane to gain focus overwrites.

Consumers: the shell-level `BottomBar` reads `focused?.wsRef` (replacing the removed `wsRef` prop); the focused pane's `TerminalClient` reads `composeOpen` and gates `<ComposeBuffer>` rendering on `isFocused && composeOpen` (board route) or `composeOpen` (single-terminal route). `composeOpen` persists across route navigation by design — the user dismisses explicitly via Escape. The compose target is frozen at open-time via `compose-buffer.tsx`'s existing wsRef snapshot on mount; cycling focus while compose is open does NOT retarget. When `focused === null` (e.g., Dashboard route with no terminal), BottomBar input handlers natural-no-op via the existing `wsRef.current?.readyState !== OPEN` guard.

**SessionProvider** (`app/frontend/src/contexts/session-context.tsx`) — root-mounted multi-server React Context. One `SessionProvider` instance lives above `<Outlet />` in `RootWrapper` and is shared by every route (`/`, `/$server/...`, `/board/$name`); it does NOT remount on server-route switches. State is keyed per server: `sessionsByServer: Map<string, ProjectSession[]>`, `sessionOrderByServer: Map<string, string[]>`, `isConnectedByServer: Map<string, boolean>`, `metricsByServer: Map<string, MetricsSnapshot | null>`. `currentServer: string | null` is dispatched from the matched route via `useMatches()` — `params.server` for `/$server/...`, `null` for `/board/$name` and `/`. Consumers read the per-server slice they need via `ctx.sessionsByServer.get(name)`; there is no transitional accessor — all consumers consume the keyed shape directly. Provider tree order: `ThemeProvider > ChromeProvider > RootWrapper(SessionProvider > OptimisticProvider > Outlet)`. The provider mirrors `currentServer`'s connection state into `ChromeProvider` so existing chrome consumers (top-bar connection dot) keep working unchanged. Persists most-recently active server to `localStorage` key `runkit-server` (single value, not split per server).

**Per-server EventSource pool** — the provider owns one `EventSource` per attached server, all connecting to `/api/sessions/stream?server=<name>`. The current server is auto-attached; non-current servers attach lazily via the `attachServer(name)` API exposed on context. Each connection updates only its own slice in the per-server maps. When a server disappears from `/api/servers`, its `EventSource` is closed and its entries are removed from every per-server map. A 3-second disconnect debounce runs per server — an `onerror` on one server does not flip another server's `isConnectedByServer` value. In practice today, `useBoards` and `useWindowPins` call `attachServer` for every known server when the sidebar mounts, so the lazy-attach guard mainly bounds open connections in routes where neither hook is mounted.

**`subscribeBoardChange`** (`session-context.tsx`) — context API that lets any consumer register a `(server: string) => void` handler and receive cross-server `board-changed` SSE events through the existing pool. `useBoards` and `useWindowPins` use it to react to per-server board mutations without opening their own `EventSource` instances — the multi-server pool already streams `board-changed`, so multiplexing through one subscriber API keeps the connection count bounded.

**Kill-redirect gate (`currentWindowEverSeen`)** — The "session/window gone" redirect in `app/frontend/src/app.tsx` (around the `useEffect` that calls `computeKillRedirect` from `app/frontend/src/lib/navigation.ts`) MUST NOT fire for a freshly-navigated URL whose (session, window) pair has not yet been observed as valid in the SSE `sessions` stream. `computeKillRedirect` accepts a `currentWindowEverSeen?: boolean` param (defaults to `true` for legacy callers and unit tests) and short-circuits to `null` when false — bypassing both the "session gone → dashboard" and "window gone → nearest sibling" branches. The caller maintains two refs: `currentWindowEverSeenRef` (boolean, flipped `true` whenever the memoized `currentWindow` is non-null) and `lastObservedUrlKeyRef` (string, `${server}|${session}|${window}`). Any URL-key change resets the seen flag to `false`. This gates two transient SSE states that are not proof the URL is unreachable: (a) a stale cached first payload that replays an older server snapshot before delivering current data, and (b) a session whose `windows` array is briefly empty while tmux enumeration propagates into the next poll tick. Without the gate, either state makes `currentSession` or `currentWindow` null on the first SSE event after navigation, the redirect fires, and `TerminalClient` unmounts mid-initialization — surfacing as an e2e flake where `.xterm-screen` briefly appears then disappears. Genuine kills from another client still redirect normally, because the URL key only resets on navigation, not on SSE payload changes: once the seen flag is `true`, a subsequent payload that drops the window triggers the redirect as before.

**Window state (Zustand)** — window optimistic state is managed by a Zustand store at `app/frontend/src/store/window-store.ts`, keyed by the immutable tmux `windowId` (`@N`). `zustand` is a production dependency in `app/frontend/package.json`. The store tracks `killed` and `pendingName` overrides per window ID, and ghost windows awaiting SSE confirmation. `AppShell` drives `setWindowsForSession` from SSE `rawSessions`. `OptimisticContext` retains session and server optimistic state only (name-based keys, not index-based). See ui-patterns.md "Zustand Window Store" section for full shape and action surface.

**`WindowInfo` type** (`app/frontend/src/types.ts`) — includes `windowId: string` (non-optional) as the immutable tmux `@N` identifier, alongside `index: number` (tmux ordering, display-only, not used as a store key). **`MetricsSnapshot` type** (`app/frontend/src/types.ts`) — mirrors the backend `metrics.MetricsSnapshot` struct. Fields: `hostname` (string), `cpu` (`{ samples: number[], current: number, cores: number }`), `memory` (`{ used: number, total: number }` bytes), `load` (`{ avg1, avg5, avg15: number, cpus: number }`), `disk` (`{ used: number, total: number }` bytes), `uptime` (number, seconds). **BottomBar** (`app/frontend/src/components/bottom-bar.tsx`) — rendered at shell level via `<Shell>`'s `bottombar` grid area on every route that mounts `<Shell>` (single-terminal, board, dashboard). Width tracks the right grid column (page width minus sidebar). Single row of `<kbd>` buttons: modifier toggles (Ctrl/Alt with sticky armed state — Cmd removed), Esc, Tab, arrow keys, Fn dropdown (F1-F12, PgUp/PgDn, Home/End). When `focused === null` (Dashboard route, no terminal), input handlers natural-no-op via the existing `wsRef.current?.readyState !== OPEN` guard — no error, no toast.

**ComposeBuffer** (`app/frontend/src/components/compose-buffer.tsx`) — native `<textarea>` overlay triggered by the compose button or file upload. Supports iOS dictation, autocorrect, paste, multiline. Send button (or Cmd/Ctrl+Enter) transmits entire text as a single WebSocket message. Terminal dims (`opacity-50`) while compose is open. Escape dismisses without sending. Accepts optional `initialText` prop for pre-populating with uploaded file paths; appends on subsequent updates while open.

**iOS Keyboard Support** — `useVisualViewport` hook (`app/frontend/src/hooks/use-visual-viewport.ts`) listens to both `resize` and `scroll` events on `window.visualViewport`, setting `--app-height` CSS custom property from `visualViewport.height`. In the single-view model (fullbleed always on), `globals.css` applies `position: fixed; inset: 0; height: var(--app-height, 100vh)` to the app shell. The bottom bar stays pinned above the keyboard; the terminal shrinks via `flex-1` and xterm refits via `ResizeObserver`.

**iOS Touch Scroll Prevention** — Fullbleed is always active in the single-view model. The `fullbleed` class is added to `<html>` by `useVisualViewport` on mount (also present as a static default in `index.html` for FOUC prevention). `globals.css` applies `overflow: hidden` and `overscroll-behavior: none` to both `html` and `body` via `html.fullbleed` selectors, preventing iOS Safari elastic bounce scrolling. The terminal container div uses `touch-none` (`touch-action: none`) so the browser yields touch gestures to xterm.js for scrollback handling.

**Browser title** — `useBrowserTitle` hook sets `document.title` dynamically based on hostname (fetched once from `GET /api/health` on app init) and route params. Dashboard: `RunKit — {hostname}`. Terminal: `{session}/{window} — {hostname}`. Omits hostname suffix when empty. Static `<title>RunKit</title>` in `index.html` remains as pre-hydration fallback.

Single-view model: there are no page transitions or per-page chrome injection. The chrome reads the current selection and renders directly.

## Boards Feature

Pane boards are named, cross-server collections of pinned tmux windows rendered as a horizontal pane dashboard at `/board/$name`. Storage lives in `@rk_board` (server-scoped tmux user-option) — see `tmux-sessions.md` § `@rk_board` for the full storage contract, value format, order keys, and stale-cleanup rules.

### Route Placement

| Route | View | Layout |
|-------|------|--------|
| `/board/$name` | Cross-server pane dashboard | Wrapped in the shared `<Shell>` grid (same wrapper as AppShell). Sidebar grid area = unified `<Sidebar currentServer={null}>`; topbar = `<TopBar mode="board">`; content = `DesktopRow`/`MobileCarousel`; bottombar = shared `<BottomBar>` (new on this route) |

`/board/$name` is registered as a child of `rootRoute` in `app/frontend/src/router.tsx`, peer to `serverLayoutRoute`. It does NOT mount AppShell — boards aggregate windows across servers, so AppShell's single-server chrome semantics (server-scoped breadcrumbs, server-scoped command-palette actions) do not apply. As of `260509-17m3-rotated-shell-layout`, BoardPage shares the AppShell `<Shell>` grid wrapper and renders `<TopBar mode="board">` + `<BottomBar>` in the shell's named areas — the route reuses the chrome topology while routing different breadcrumb / pane-area content into it. BoardPage retains its own `<CommandPalette>` mount because board-route-only entries (Switch / Leave Board View / Cycle Pane Focus) need a registration site, and the AppShell palette doesn't mount on `/board/<name>`. `SessionProvider` lives at the root and exposes per-server state, so the board route renders `<Sidebar currentServer={null} ...>` directly with no special-case. The Sidebar's per-server session groups + Boards section serve discovery and entry from any route. See ui-patterns.md § Boards View for the layout.

### Backend Surface

| Layer | File | Responsibility |
|-------|------|---------------|
| Storage | `app/backend/internal/tmux/board.go` | `@rk_board` parser/serializer, `Pin`/`Unpin`/`Reorder`/`GetBoard`/`ListBoards`, `ComputeOrderKey` (lowercase-`a..z` fractional indexing), `RemoveAllByWindowID` (used by SSE eager cleanup), validators (`ValidBoardName`, `ValidWindowID`, `ValidOrderKey`) |
| HTTP handlers | `app/backend/api/boards.go` | `handleBoardsList`, `handleBoardGet`, `handleBoardPin`, `handleBoardUnpin`, `handleBoardReorder` — register in `router.go` `buildRouter()` |
| Interface | `app/backend/api/router.go` `TmuxOps` | `ListBoards`, `GetBoard`, `PinBoard`, `UnpinBoard`, `ReorderBoard` (production wires `prodTmuxOps` to `tmux.*`; tests use `mockTmuxOps`) |
| SSE | `app/backend/api/sse.go` | `BoardEntriesFetcher` interface (parallel to `OrderFetcher`), `previousBoardJSON` per-server cache, `broadcastBoardChanged(server, payload)`, `event: board-changed` (kebab-case, parity with `event: session-order`), bootstrap-on-first-poll, cached-snapshot send to new clients (between `session-order` and `metrics`), and window-kill detection that calls `RemoveAllByWindowID` and broadcasts one `cleanup` event per affected board |

### Constitution Alignment

- **II (No Database)** — pin state lives only in tmux server-scoped options + browser `localStorage` for view-local pane widths
- **IV (Minimal Surface Area)** — adds exactly one route (`/board/$name`); no settings or admin pages
- **V (Keyboard-First)** — pin/unpin/switch/cycle reachable via Cmd+K palette; pane focus cycle via Cmd+[/]
- **VI (Tmux Sessions Survive Server Restarts)** — pin state persists with the tmux server; rk-go restart is bridged by the SSE bootstrap; `tmux kill-server` loses the boards on that server (expected and acceptable)
- **VII (Convention Over Configuration)** — no new env vars, no new config files; reuses the established `@rk_*` user-option pattern

## Design Decisions

- **Go backend + Vite SPA over Next.js monolith** — decouples frontend and backend for independent iteration. Go backend is a stable, long-lived API that outlives any individual frontend. Multi-client API support (web, mobile, CLI) without split API surface
- **Single port architecture** — Go serves API, WebSocket relay, and SPA static files on one port. The two-port split (Next.js :3000, relay :3001) was a Node.js artifact — separate processes required separate ports. Go serves everything in one binary
- **chi over stdlib ServeMux** — chi for middleware chaining (CORS, logging, recovery). Go 1.22+ ServeMux has pattern matching but lacks ergonomic middleware composition
- **TanStack Router over React Router** — type-safe params and search params, built-in loader pattern. Single route `/:session/:window` in the new frontend
- **Vite proxy in dev (not CORS)** — single browser URL, no CORS config needed. WebSocket upgrade works transparently. Go includes chi CORS middleware for production/non-browser clients
- **SPA fallback in Go** — Go serves standalone; TLS termination handled externally via Tailscale Serve when needed
- **SSE (not WebSocket) for session state** — simpler, server-push only, naturally resilient. Module-level hub deduplicates polling across tabs (one `FetchSessions()` per server per interval). Clients indexed by server name (`map[string][]*sseClient`) — broadcast iterates only the target server's slice. `sync.RWMutex` with `RLock` for read-only poll operations, `Lock` for writes. SSE data includes `isActiveWindow` per window, enabling UI sync when users switch tmux/byobu windows via terminal shortcuts
- **Full snapshots (not diffs)** — small payload (<100 sessions), simple client logic
- **Independent panes per browser client** — no cursor fights, agent pane untouched. The relay pty follows tmux window switches natively (runs `tmux -L runkit attach-session`)
- **Every tmux session is a project** — no config, no "Other" bucket. Project root derived from window 0's `pane_current_path`
- **Config via env vars (not YAML)** — `.env` committed with defaults (`RK_PORT`, `RK_HOST`), `.env.local` for overrides, loaded via `.envrc` (direnv). Go and Vite read `RK_PORT`/`RK_HOST` directly. `dev.sh` translates `RK_PORT+1` for the backend subprocess. No relay port — single port serves everything
- **Dedicated tmux server (`-L runkit`)** — run-kit sessions live on a named tmux server `runkit` with its own config (`internal/tmux/tmux.conf` loaded via `-f`). The default tmux server is also queried for session discovery (read-only display of external sessions). This replaces byobu integration: `CreateSession()` uses plain `tmux new-session` on the runkit server (no byobu dependency). The `runkit` server provides isolation, a custom status bar matching run-kit's dark theme, and byobu-style F-key keybindings (F2/F3/F4). `SessionInfo` carries a `Name` field; `ProjectSession` has `Name` and `Windows` fields. The caller passes the target server name to each tmux operation
- **Multi-server session enumeration** — `ListSessions(ctx, server)` queries the specified tmux server. The API layer calls it once per server (e.g., `runkit` and `default`), merging results. Session-group filtering applies to both servers. See `docs/memory/run-kit/tmux-sessions.md`
- **Derived chrome (not slot injection)** — Single-view model means only one chrome state (terminal-focused). Top bar and bottom bar derive content from the current session:window selection. No `setLine2Left`/`setLine2Right`/`setBottomBar` setters. Split React Context preserved for performance (state vs dispatch).
- **Root-mounted multi-server SessionProvider with per-server EventSource pool** — One `SessionProvider` instance lives at the root above `<Outlet />` (in `RootWrapper`) and is shared by every route. State is keyed per server (`Map<string, T>`); `currentServer: string | null` is dispatched from the matched route via `useMatches()`. The pool opens one `EventSource` per server lazily — current server eager, others via `attachServer(name)`. Provider survives server-route switches (no remount, no state rebuild) and lets the Board route render the same unified `<Sidebar>` as AppShell with `currentServer === null`. Cross-server `board-changed` events are multiplexed through `subscribeBoardChange` so cross-server hooks (`useBoards`, `useWindowPins`) don't open separate connections. Rejected: per-route providers bridged via a sibling cross-server context — adds two contexts, double-fetches `/api/servers`, doesn't solve BoardPage sidebar reuse cleanly. Rejected: `Record<string, T>` instead of `Map<string, T>` — Maps give clear "server unknown" vs "server known with empty value" semantics. (`260508-dc0t-multiserver-session-provider`)
- **Single-view layout (sidebar + terminal/dashboard) replaces three pages** — Dashboard and Project page functionality subsumed by the sidebar + Dashboard view. Terminal is the main content on `/:session/:window`; Dashboard renders on `/`. No page transitions.
- **Dashboard as inline component in app.tsx (not separate route layout)** — The Dashboard renders in the terminal area via a conditional branch (`sessionName && windowIndex ? <TerminalClient/> : <Dashboard/>`). This reuses the existing chrome skeleton (top bar, sidebar) and avoids duplicating the app shell layout. Rejected: separate route with dedicated layout — would require lifting sidebar/chrome into a shared layout component and restructuring the route tree.
- **POST-only API client with path-based intent** — Each mutation is a separate function with its own URL (e.g., `killSession(session)` → `POST /api/sessions/:session/kill`). No multiplexed `action` field in request bodies.
- **Sidebar + drawer pattern on mobile** — Desktop sidebar is drag-resizable (default 220px, min 160, max 400, localStorage persist), collapsible. Mobile (< 768px) uses a left-side drawer overlay triggered by `☰`. Preserves session/window tree layout across breakpoints.
- **Prebuilt binaries over build-from-source Homebrew formula** — ship cross-compiled binaries via GitHub Release. Zero build dependencies on user's machine (no Go, Node.js, pnpm). Faster install. Rejected: build-from-source formula (like tu) which requires all build tools. (`260317-ukyz-homebrew-deployment`)
- **`embed.FS` with copy step over restructuring repo** — copy `app/frontend/dist/` into `app/backend/build/frontend/` at build time because Go's `//go:embed` cannot reference files outside the package directory. Simple build-time operation. Rejected: colocating frontend output with Go source (breaks dev workflow). (`260317-ukyz-homebrew-deployment`)
- **VERSION file + ldflags over Go constant** — version sourced from `VERSION` file, injected via `-X main.version=...`. Shell scripts can read/write plain text. No code changes for version bumps. Rejected: Go constant (requires code change per release). (`260317-ukyz-homebrew-deployment`)
- **Cobra over custom CLI parsing** — industry standard, auto-generates help, 5 subcommands is the sweet spot. Rejected: stdlib `flag` (no subcommand support), custom parsing (unnecessary). (`260317-ukyz-homebrew-deployment`)
- **SPA handler dual-mode (embedded FS vs filesystem)** — `hasEmbeddedAssets()` checks whether `frontend/` contains more than `.gitkeep`. Production uses `embed.FS`, dev uses filesystem (Vite handles frontend). Automatic detection, no build tags or env vars needed. (`260317-ukyz-homebrew-deployment`)
- **Active window sync via `history.replaceState` (not `router.replace()`)** — When byobu switches windows, the terminal relay pty already shows the correct content. The UI syncs breadcrumb, URL, and action targets via SSE polling (2.5s). URL updates use `window.history.replaceState()` which is invisible to the router — no re-render, no terminal reinitialization.
- **Sticky modifier state via useRef + forceUpdate** — `useModifierState` uses a ref for the authoritative state and a counter state to trigger re-renders. Ensures `consume()` reads the latest value atomically without stale closure issues.
- **Compose buffer as native textarea (not xterm input)** — xterm renders to `<canvas>`, blocking OS-level input features. The compose buffer provides a real `<textarea>` where dictation, autocorrect, paste, and IME all work. Text sent as a single WebSocket message.
- **Armed modifiers bridge to physical keyboard** — When bottom-bar modifiers (Ctrl/Alt) are armed, a capture-phase `keydown` listener intercepts physical keypresses, translates them to terminal escape sequences (Ctrl+letter → control characters, Alt → ESC prefix), and sends via WebSocket. Prevents xterm from receiving the unmodified key. Ignores real Cmd/Ctrl/Alt held by the OS.
- **File upload via server filesystem (not terminal binary injection)** — Browser uploads file to `POST /api/sessions/:session/upload`, server writes to `.uploads/` in project root, path auto-inserted into compose buffer. Works because run-kit server and tmux are always co-located; the browser is the remote part. Session identified by URL param (consistent with other session-scoped endpoints, replaces legacy form field approach)
- **Handler files split by resource domain (not monolithic routes.go)** — Each handler file owns one resource: `sessions.go`, `windows.go`, `directories.go`, `upload.go`, `sse.go`, `relay.go`, `spa.go`, `health.go`. `router.go` owns middleware, dependency interfaces, and route registration only. (`260312-r4t9-go-backend-api`)
- **Dependency injection via interfaces for handler testability** — `Server` struct holds `SessionFetcher` and `TmuxOps` interfaces, plus `hostname` (computed via `os.Hostname()` in `NewRouter()`, empty string on failure) and `*metrics.Collector` (created and started in `NewRouter()` with the server context). `TmuxOps` interface includes `SetWindowOption(ctx, session, index, server, option, value)` and `CreateWindowWithOptions(session, name, cwd, server, options)` for iframe window metadata operations. `NewRouter()` wires production implementations; `NewTestRouter()` accepts mocks (including hostname). Enables `httptest.NewRecorder` tests without live tmux. (`260312-r4t9-go-backend-api`)
- **Hostname via health endpoint (not a dedicated endpoint)** — hostname is an OS-level value computed once at startup and stored in `Server` struct. Exposed via `/api/health` response (adding a field to an existing endpoint) rather than a new `GET /api/hostname` route. Minimal surface area, consistent with the single-port architecture. (`260320-uq0k-hostname-browser-title`)
- **Per-window fab enrichment via `fab pane map` (replaces per-session file reading)** — Single `fab pane map --json --all-sessions` subprocess call per SSE tick replaces per-session `.fab-status.yaml` + `.fab-runtime.yaml` file reads. Provides per-window resolution (each worktree window shows its own change/stage) instead of per-session (all windows inherited session-level state). Decouples from internal file formats. `internal/fab` package deleted entirely. The subprocess invokes the `fab` router on PATH (not a hardcoded kit binary path) with `cmd.Dir = repoRoot` so the router can resolve the project's `fab_version` from `fab/project/config.yaml`. (`260313-3vlx-pane-map-enrichment` introduced pane-map enrichment via the then-in-repo `fab-go` binary; `260417-5nok-fix-sidebar-fab-state` migrated to the router after the binary was removed from the repo. Supersedes `260312-r4t9-go-backend-api` and `260313-txna-rich-sidebar-window-status` decisions)
- **Backend settings file over localStorage-only** — `~/.rk/settings.yaml` via `internal/settings/` package + `GET/PUT /api/settings/theme` endpoints. Survives browser cache clears, works across devices accessing the same server. localStorage kept as synchronous cache for instant reads before API responds on page load. Simple `key: value` text parsing (not yaml.v3) — one field currently, avoids re-adding a heavyweight dependency. Settings are global (not per-server, no `?server=` param). (`260323-7wys-ansi-palette-theme-rework`)
- **Full ANSI palette over minimal color set** — 22 canonical terminal colors per theme enables three consumers from one source: CSS (8 derived colors via `deriveUIColors`), xterm.js (full 22-color theme via `deriveXtermTheme`), and tmux (auto-themed via ANSI colour indices in static tmux.conf). Rejected: 8-color model with separate xterm colors (duplicative, inconsistent). (`260323-7wys-ansi-palette-theme-rework`)
- **Static tmux.conf with ANSI indices over runtime `tmux set -g`** — tmux.conf uses `colour0`-`colour15` indices. Changing xterm.js palette auto-themes tmux chrome (status bar, pane borders, pane-border-format). Zero backend involvement for theme switching. Rejected: runtime tmux commands (complex, fragile, requires backend communication). (`260323-7wys-ansi-palette-theme-rework`)
- **`run-kit.yaml` at git root for per-project preferences (not env vars or server config)** — `config.FindGitRoot()` walks up from `pane_current_path` to locate the git root, then reads/writes `run-kit.yaml` there. Session color is a per-project user preference, not server configuration (which uses env vars per constitution). Previous `run-kit.yaml` (removed 2026-03-13) stored port/host config — this reintroduction serves a different purpose. File is gitignored by convention. Simple line-by-line parsing (no yaml.v3 dependency). (`260416-jn4h-session-window-color-tinting`)
- **Dual storage: `run-kit.yaml` for session color + tmux `@color` for window color** — Session colors need persistence across tmux restarts (file-based). Window colors are ephemeral and tmux-native (user options survive session lifetime but not server restart). Rejected: all in `run-kit.yaml` (would require tracking per-window state in a file, complex synchronization with tmux window lifecycle). (`260416-jn4h-session-window-color-tinting`)
- **Separate SSE event type for metrics (`event: metrics`) over embedding in sessions payload** — Metrics are server-wide (one payload for all clients), while sessions are per-tmux-server and deduplicated. CPU samples change every tick (dedup logically impossible). Separation avoids inflating session payloads and allows independent evolution. Rejected: embedding in sessions payload (sent N times per server, complicates dedup). (`260411-z63r-sidebar-host-window-panels`)
- **Server-side ring buffer with full history per tick over client-side accumulation** — New SSE clients immediately get complete 60-sample CPU sparkline history. No reconnection logic needed to rebuild history. At 60 float64s, ~500 bytes per tick is negligible. Rejected: client-side accumulation from deltas (reconnection complexity, frontend state management). (`260411-z63r-sidebar-host-window-panels`)
- **Path-based reverse proxy over subdomain routing** — `/proxy/{port}/*` keeps all traffic on one origin, avoiding CORS complexity and DNS configuration. Per-port `ReverseProxy` instances cached in `sync.Map` (read-heavy, single-digit entry count). `ModifyResponse` rewrites localhost/127.0.0.1 URLs in HTML `src`/`href`/`action` attributes to proxy paths; handles gzip decompression. JS-constructed URLs are not rewritten (acceptable v1 gap). Rejected: subdomain routing (requires DNS/hosts setup), service worker injection (fragile). (`260416-6b0h-iframe-proxy-windows`)
- **Tmux user-defined options for iframe state over database/config file** — `@rk_type` and `@rk_url` as tmux window options preserve constitution principles II (no database) and VI (sessions survive restarts). State is co-located with the tmux window lifecycle — kill the window, lose the metadata. Reactive via existing SSE polling (no new transport). Rejected: database/config file (violates constitution), window name encoding (fragile). (`260416-6b0h-iframe-proxy-windows`)
- **Extend existing create endpoint with optional rkType/rkUrl over new endpoint** — Adding optional `rkType`/`rkUrl` to `POST /api/sessions/{session}/windows` reuses the existing creation flow. When `rkType` is present, uses `CreateWindowWithOptions` for atomic `\;`-chained tmux command (prevents SSE seeing window before metadata set). When absent, behavior unchanged (standard terminal window). Rejected: separate `POST .../windows/iframe` endpoint (unnecessary surface area). (`260416-6b0h-iframe-proxy-windows`)
- **procfs direct reading over external monitoring tools** — `internal/metrics` reads `/proc/stat`, `/proc/meminfo`, `/proc/loadavg`, `/proc/uptime` directly. No dependency on `top`, `htop`, or monitoring agents. `syscall.Statfs` for disk. All readers return zero values on non-Linux (macOS dev graceful degradation). No new Go dependencies. (`260411-z63r-sidebar-host-window-panels`)
- **`currentWindowEverSeen` gate on kill-redirect over unconditional "gone → redirect"** — The redirect in `app.tsx` that fires when `currentSession` or `currentWindow` is null treats every SSE payload as authoritative by default, but the first payload after a navigation is not: the SSE hub can replay a cached snapshot before delivering fresh data, and tmux window-list enumeration can lag by one poll tick when a session was just created. Unconditionally redirecting on either transient state tears down `TerminalClient` mid-mount. The gate (a ref keyed on `${server}|${session}|${window}`, flipped true the first time `currentWindow` resolves non-null, reset on URL change) defers the redirect until the URL target has been observed as valid at least once. Rejected: inspecting SSE payload identity or delaying `isConnected` (both push complexity into the hub layer and would not distinguish real kills from stale catch-up). Rejected: adding a fixed startup debounce (arbitrary delay, races with fast tests). The gate preserves fast-redirect behavior for genuine kills (seen flag stays true after first valid observation) while eliminating the startup race. (`260417-g9uo-mobile-tap-focus-regression`)
- **CSS Grid topology for the rotated shell over nested flex** — The shell rotation makes the sidebar a column that spans all three rows (topbar, content, bottombar). Expressing "sidebar full-height; right column is a 3-row stack" with nested flex requires either a sibling sidebar with explicit height management plus a flex-col right column, or a 2-row layout with sidebar nested inside row 1 — neither matches the visual model and both diverge as future cells are added. Grid encodes the topology directly via `grid-template-areas: "sidebar topbar" "sidebar content" "sidebar bottombar"`; collapse becomes a one-line column-width change (`${sidebarWidth}px 1fr` ↔ `0 1fr`) animatable as a single `grid-template-columns` transition. Modern browsers (Safari 14+, Chrome 100+) animate `grid-template-columns` reliably. (`260509-17m3-rotated-shell-layout`)
- **Sidebar collapses to 0px, not a 48px rail** — Maximum content width when collapsed, accepting a small visual layout shift on toggle (the hamburger appears to "move" from sidebar-adjacent to page-adjacent, though structurally it always lives at TopBar.left — see next decision). Rejected: a 48px icon rail (VS Code/Linear/Slack convention) — avoids the visual relocation but consumes content width for a usability gain that diverges from the user's stated preference for max content width. (`260509-17m3-rotated-shell-layout`)
- **Hamburger statically rendered at TopBar.left** — Makes the visual relocation on collapse a free side effect of the grid column collapsing — no component-level relocation logic, no conditional mount/unmount, no animation orchestration. The hamburger's home is always TopBar.left; the column is what changes. Rejected: state-conditional placement (hamburger inside sidebar when open, inside topbar when closed) — adds a state-dependent component-tree mutation that complicates focus management and adds animation responsibility. (`260509-17m3-rotated-shell-layout`)
- **Single `sidebarOpen` boolean over `sidebarOpen + drawerOpen`** — The pre-rotation distinction between "sidebar open on desktop" and "drawer open on mobile" was a rendering-form difference, not a state difference. The user's intent ("show me the sidebar") is one boolean; the renderer picks the form (grid column vs absolute overlay) by viewport width. Collapsing the two booleans simplifies persistence (one localStorage key, `runkit-sidebar-open`), removes a class of "they're out of sync" bugs, and makes the keyboard chord a single state mutation. Rejected: keep two booleans — preserves a historical artifact at the cost of doubled persistence/reset logic and a confusing surface for new contributors. (`260509-17m3-rotated-shell-layout`)
- **`<Shell>` as a shared wrapper over per-route shell components** — Both AppShell and BoardPage need the identical 3-row × 2-column grid, sidebar collapse, mobile overlay, and visual-viewport handling. A single `<Shell>` (`src/components/shell/shell.tsx`) lets both routes consume a uniform topology by wrapping children in `style={{ gridArea: "sidebar" | "topbar" | "content" | "bottombar" }}` elements; collapse animation, mobile overlay, and the topbar/content/bottombar geometry stay in one place. Rejected: per-route shell components — guaranteed to diverge over time (different collapse timings, drift in mobile overlay handling), defeats the rotation's "one topology, many routes" goal. Pattern note: future routes that need the same chrome should adopt `<Shell>` rather than reimplementing the grid. (`260509-17m3-rotated-shell-layout`)
- **`FocusedTerminalContext` as a separate, dedicated context (not a slice of ChromeContext)** — When BottomBar moved up to shell-level, it needed a way to target the focused terminal regardless of route. Folding the focused-terminal target into ChromeContext would mix UI state (sidebar open, fixed-width) with terminal-WS state and force every Chrome consumer to re-render whenever focus changes. A small dedicated context with its own provider (`FocusedTerminalProvider` mounted in `RootWrapper`) keeps the dependency narrow: only BottomBar and the ComposeBuffer-rendering TerminalClient subscribe. Compose state (`composeOpen` / `setComposeOpen`) was lifted to the same context so the shell-level BottomBar can open compose for the focused pane on the board route — alternative was lifting `<ComposeBuffer>` itself to shell level, which would have required rewiring the file-upload chain (`composeFiles`/`composeInitialText`/`useFileUpload`) across 3-4 components. Frozen-target semantics are preserved because `compose-buffer.tsx` already snapshots `wsRef` on mount; only the focused pane's `TerminalClient` renders ComposeBuffer at any time. Pattern note: future multi-input UIs that need a single shared input target should follow this "lift focused-input target to context" pattern rather than prop-drilling. (`260509-17m3-rotated-shell-layout`)
- **Read-only setter pair for in-memory + persisted state (`setSidebarWidth` + `persistSidebarWidth`)** — Drag-resize fires per-pointermove (potentially 60+ events per second). Persisting on every event would write localStorage 60 times per second per drag — unnecessary I/O and cache thrash. Splitting the dispatch into `setSidebarWidth(width)` (in-memory state) and `persistSidebarWidth(width)` (localStorage write) lets the drag handler call the cheap one in the loop and the expensive one once at drag-end. Convention note: future state with the same shape (live in-memory updates + final commit to persistence) should follow this two-setter pattern rather than debouncing inside the dispatch (which couples state lifecycle to a timing heuristic). (`260509-17m3-rotated-shell-layout`)
- **Mobile sidebar overlay positioned via grid (`gridRow: "2/4"`), not `fixed inset-0`** — The spec's literal phrasing was `fixed inset-y-0 left-0 z-50`, but the project convention recorded in `fab/project/context.md` and the existing `mobile-layout.spec.ts` assertion (`sidebarBox.y > 0`) requires the mobile overlay to leave the topbar visible — the topbar has the hamburger which closes the overlay, so covering it would be a usability regression. The implementation uses a grid-row-spanning child with `position: absolute` (not `fixed`) so the overlay anchors below the topbar inside the Shell. Pragmatic resolution: spec-wording "fixed-position overlay with backdrop" is satisfied — the overlay is fixed-position-style (absolute inside the Shell that itself is full-viewport), it has a backdrop, it carries `role="dialog" aria-modal="true"`. (`260509-17m3-rotated-shell-layout`)
- **Known follow-up — `sidebarChildren` prop duplication on `<Shell>`** — Both `AppShell` and `BoardPage` build the sidebar JSX once into a local `sidebarElement`, then pass it as `<Shell sidebarChildren={...}>` (used on mobile overlay) AND wrap it in a desktop-branch `<aside style={{gridArea:"sidebar"}}>{sidebarElement}</aside>` (used on desktop). This duplicates a small bit of JSX at every Shell consumer. Considered alternatives: a render-prop or a single named "sidebar" slot owned by Shell. The desktop branch in AppShell also includes the drag-handle aside wrapper, which BoardPage does NOT need — collapsing the duplication requires either (a) Shell knowing about the drag handle (small abstraction leak) or (b) exposing a separate "desktopChrome" slot. Captured here as a deferred refactor — not in 17m3's scope. (`260509-17m3-rotated-shell-layout`)

## Security

- All subprocess calls use `os/exec.CommandContext` with argument slices (never `sh -c` or shell strings)
- All `exec.CommandContext` calls include timeout via `context.WithTimeout` (10s tmux, 30s build). Terminal relay attach uses `context.WithCancel` (long-lived, cancelled on disconnect)
- User input validated via `internal/validate` before reaching any subprocess
- Directory listing restricted to `$HOME` via `ExpandTilde()` — rejects `..` traversal, absolute paths outside home, and `~username` syntax. Symlinks under `$HOME` are not resolved (accepted risk for local dev tool)
- File uploads: filename sanitized via `SanitizeFilename()` (strips path separators, null bytes, leading dots, collapses dot sequences); 50MB size limit enforced server-side via `http.MaxBytesReader`; writes via `os.Create` (not subprocess)
- CORS: permissive by default (`*` origin) for multi-client API flexibility. TLS handled by Tailscale Serve in production
